# FlowX 桌面端框架设计文档

> 基于实际代码实现，版本 v1.0。FlowX 是一个多平台内容发布桌面客户端，支持抖音/小红书/快手账号管理与一键发布。

---

## 一、技术选型与架构概述

### 1.1 技术栈

### 1.3 核心设计原则

| 原则 | 描述 |
|:---|:---|
| **账号隔离** | 每个账号使用独立的 Electron `session` partition（`persist:account_{id}`），天然隔离登录态，避免平台互相踢下线 |
| **适配器模式** | 平台差异抽象为 `StandardPublishConfig`，共享 `runStandardPublish` 流程。修改一个平台不会影响其他平台 |
| **导航稳定性优先** | 所有 DOM 操作前检查 `isLoading` + 导航静默期，避免"页面正在跳转时执行 JS 导致 frame disposed" |
| **崩溃自动恢复** | 发布窗口的渲染进程崩溃（`render-process-gone`）后自动 reload 1 次（Electron 31 已从根源解决，此处为兜底） |
| **分层可见性** | `AccountCredential`（含 cookie）仅在主进程存在，IPC 返给前端的是脱敏的 `AccountInfo` |
| **IPC 安全** | 渲染层通过 preload 暴露的白名单 API 与主进程通信；所有外部 URL 的窗口不接收 IPC 消息 |

## 三、账号服务（AccountService）

### 3.1 授权流程（核心）

用户在 AccountPanel.vue 点击"新增账号" → 选择平台 → 开始授权：

```
startAuth(platform)
  └─> ipc account:beginAuth
        └─> AccountService.beginAuthorization(platform)
              │
              │ [STEP 1] 账号窗口创建
              │   new BrowserWindow({ partition: `persist:account_${id}`, ... })
              │   loadURL(platform.authUrl)
              │
              │ [STEP 2] 监听保存信号（三重机制）
              │   ① page-title-updated 事件（页面标题变化 → 可能已登录）
              │   ② did-navigate-in-page hash 变化（SPA 路由）
              │   ③ 500ms 轮询 executeJavaScript 检测登录关键字
              │
              │ [STEP 3] 检测登录态（调用平台适配器 detectLoggedIn）
              │   最多重试 2 次，等待 2s 后检测
              │
              │ [STEP 4] DOM 信息提取
              │   nickname / avatar / platformAccountId / fansCount / followCount / likeCount
              │   由平台适配器 extractPageInfo 实现（详见 platforms/xiaohongshu.ts §3.2）
              │
              │ [STEP 5] cookie 提取 + 校验
              │   ses.cookies.get({ url: undefined }) 提取所有 cookie
              │   ▸ 黑/白名单双重校验：
              │     · 必须有真实昵称或平台账号 ID（过滤"抖音创作者中心"等占位）
              │     · cookies 数量 ≥ 5（空账号只有少量 session cookie）
              │     · 必须包含 session/token/a1/sec_user_id 等关键字段之一
              │
              │ [STEP 6] 账号去重
              │   同一 platform + platformAccountId / userId 命中 → 更新旧记录
              │   否则 → 新增记录
              │
              └─ 加密写入 SecureStore，返回 AccountInfo
```

### 3.2 Cookie 注入机制

```typescript
// injectAccountCookies(accountId, targetUrl?)
// 1. 获取目标 partition
// 2. 从 AccountCredential 读取 cookies[]
// 3. 逐个 set({ url: cookie.url 或 targetUrl, ... })
// 4. 统计 injected / skipped / failed
// 5. 跳过反自动化 cookie：acw_tc / ttwid / _csrf / _x_b / xsec_
//    （这些是平台反自动化系统标记，每次打开新浏览器窗口都会重新生成）
```

使用场景：
- **账号刷新**：先用旧 partition 打开 homeUrl，重新检测 + 提取 + 覆盖保存
- **发布前**：在发布窗口的 partition 中注入账号 cookies，使页面以已登录状态加载创作中心
- **打开创作中心**：用户手动打开某个账号的创作中心查看数据

## 四、发布引擎（PublishEngine）

### 4.1 发布任务数据流

```
Publish.vue submitPublish()
  │
  ├─> publishStore.submit(request)      // 前端登记 queued 状态
  │     └─> electronApi.submitPublish(request)
  │
  │ [主进程]
  └─> PublishEngine.submit(req)
        │
        ├─ 生成 taskId = `task_{6位随机}`
        ├─ 展开为 PublishItemProgress[]（每个 accountId 一条）
        ├─ 登记到 running Map
        ├─ ✅ 立即推送一次 status=queued 到渲染层（避免点击没反应）
        │
        ├─ 若有 scheduledAt → setTimeout(runTask, delay)
        └─ 否则立即 runTask(task)
              │
              ├─ drain() 循环：activeCount < concurrency 时执行 item
              ├─ runItem(taskId, accountId)
              │    ├─ 校验账号存在/未过期（失败直接 failed）
              │    ├─ injectAccountCookies(accountId)
              │    └─ adapter.publish(accountId, request, onProgress)
              │         └─> platforms/shared.ts runStandardPublish()
              │
              └─ 每次 onProgress 回调 → notifyStatus(task, verbose=true)
                   └─> ipc publish:statusChanged → publishStore._applyStatusUpdate
                         └─> Publish.vue 折叠面板实时刷新
```

### 4.2 并发控制

```typescript
let concurrency = 3;        // 可通过 publish:setConcurrency 动态调整（1~20）
let activeCount = 0;        // 正在执行的 item 数
let queue: Array<{ taskId, accountId }>;  // 等待队列

function drain() {
  while (activeCount < concurrency && queue.length > 0) {
    const next = queue.shift()!;
    activeCount++;
    runItem(next.taskId, next.accountId).finally(() => {
      activeCount--;
      setImmediate(drain);     // 尾递归：释放事件循环后继续调度
    });
  }
}
```

**为什么并发数默认是 3**：每个发布 item 都打开一个独立的 BrowserWindow（含完整的 Chromium 实例），内存和 CPU 占用与窗口数线性相关。3 并发在普通办公电脑（16GB RAM）上保持稳定，更多窗口会触发系统级 swap 导致所有窗口都变慢。

### 4.3 发布状态（PublishStatus）

```
queued → running → success
              → failed
              → cancelled
              → scheduled
```

- `running` 占 0~95%（上传/填写/点击各阶段递增）
- `success` = 100%，显示绿色
- `failed` 保留当前进度并显示错误消息，窗口保持打开供用户查看原因
- `scheduled` 为定时发布，时间到后自动开始

### 4.4 IPC 推送通道设计

主进程 `PublishEngine.notifyStatus` → 渲染层 `stores/publish.ts _applyStatusUpdate`，是唯一的状态同步路径。

**关键安全过滤**：在发送 IPC 前，`notifyStatus` 遍历 `BrowserWindow.getAllWindows()`，对每个 webContents 检查 URL：
- 外部 URL（`http(s)://`）→ **跳过**。把 IPC 消息发到抖音创作者中心页面会触发 `Render frame was disposed before WebFrameMain could be accessed`
- 本地 URL（`file://` / `chrome-error://`）→ 正常推送
- 发布窗口通过 `win.webContents.userAgent` + `_flowxPublishWindow` 标识位双重识别

这是防止"发布成功后主窗口卡死/白屏"的关键机制。

### 4.5 重启恢复

应用启动时 `PublishEngine.init()` 从磁盘读取 `publishTasks`（最近 200 条）。若其中存在 `queued/running` 状态的任务（说明上次应用被强制关闭），统一重置为 `failed`，保证数据一致性。

### 4.6 测试模式（Test Mode）

**设计目标**：在不真正发布内容的前提下，验证发布流程是否正常工作，用于调试平台适配器和验证表单填写逻辑。

**核心机制**：

**测试模式行为**：

| 行为 | 正式发布 | 测试模式 |
|:---|:---|:---|
| 填写表单 | ✅ 完整填写 | ✅ 完整填写 |
| 上传素材/封面 | ✅ 上传 | ✅ 上传 |
| 点击发布按钮 | ✅ 点击 | ❌ **不点击**，高亮标记 |

**关键实现（shared.ts `setupTestModeWindow`）**：

1. **拦截发布按钮**：在最终发布步骤前检查 `testMode`，若为 true 则跳过点击
2. **高亮发布按钮**：通过 JS 注入给发布按钮添加醒目的红色边框 + 闪烁动画
3. **禁用 beforeunload**：覆盖页面 `beforeunload` 事件，确保测试窗口可以正常关闭
4. **强制窗口销毁**：close 事件中调用 `win.destroy()` 兜底，防止页面挽留对话框阻塞
5. **检测表单填写状态**：在测试结束前扫描 DOM，生成 `testResult` 报告

**窗口管理流程**：

```
测试模式窗口生命周期：
  创建窗口 → 导航到发布页 → 填写表单 → （跳过发布按钮）→ 高亮按钮
                                                          ↓
                                                    生成 testResult
                                                          ↓
                                                    窗口保持打开
                                                          ↓

```

### 4.7 重试机制（Retry）

支持三种重试方式，覆盖不同场景：

| 重试方式 | 触发场景 | 行为 | IPC 通道 |
|:---|:---|:---|:---|
| **重试失败账号** | 任务有失败项 | 仅对 failed/cancelled 的账号重新执行 | `publish:retry` |
| **重新测试** | 测试状态的任务 | 对**所有账号**重新执行测试模式 | `publish:retryAsTest` |
| **立即发布** | 测试状态的任务 | 对**所有账号**转为正式发布 | `publish:retryAsPublish` |

**实现要点**：

```typescript
// 重试失败账号：过滤出失败项，复用原请求参数
async retryPublish(taskId: string): Promise<string | null> {
  const task = tasks.get(taskId);
  const failedIds = task.items.filter(i => 
    i.status === 'failed' || i.status === 'cancelled'
  ).map(i => i.accountId);
  if (failedIds.length === 0) return null;
  return submit({ ...task.request, accountIds: failedIds });
}

// 重新测试：所有账号 + testMode=true
async retryAsTest(taskId: string): Promise<string | null> {
  const task = tasks.get(taskId);
  return submit({ 
    ...task.request, 
    testMode: true,
    accountIds: task.items.map(i => i.accountId)
  });
}

// 立即发布：所有账号 + testMode=false（或不传）
async retryAsPublish(taskId: string): Promise<string | null> {
  const task = tasks.get(taskId);
  return submit({ 
    ...task.request, 
    testMode: false,
    accountIds: task.items.map(i => i.accountId)
  });
}
```

**Preload 暴露**：所有重试通道需在 `preload/index.ts` 中通过 `contextBridge` 暴露，否则前端会报 "FlowX API 缺失"。

### 5.6 Closed Shadow DOM 穿透点击（小红书发布按钮）

小红书的发布按钮是自定义 Web Component：

```html
<xhs-publish-btn>    <!-- host element (visible to JS) -->
  <!-- Shadow Root (内部 closed，JS 无法访问) -->
  #shadow-root (closed)
    <div>
      <button>发布笔记</button>   <!-- 真正的按钮 -->
    </div>
</xhs-publish-btn>
```

**为什么 JS click() 不生效**：`element.click()` 只在 host 元素上触发，但事件监听器绑定在 shadow DOM 内部的 `<button>` 上。

**CDP 解决**：使用 Chrome DevTools Protocol `DOM.getDocument(depth: -1, pierce: true)`
- `pierce:true` = 强制穿透 closed shadow DOM
- 返回结果包含 shadow DOM 中的所有节点
- 遍历找到 `<button>` 或 `<div role="button">`，计算其坐标
- 用 `Input.dispatchMouseEvent(type: mouseMoved → mousePressed → mouseReleased)` 模拟真实鼠标点击

```typescript
// shared.ts cdpClickPublishButton(win, log, keywords)
1. CDP DOM.getDocument({ depth: -1, pierce: true })   // ← 关键：pierce:true
2. 遍历树，收集:
   - button 元素（含 shadow DOM 内）
   - a 元素
   - div[role=button] / span[role=button]
   - 含关键字文本的元素
3. DOM.getBoxModel(nodeId) → 计算中心坐标 (x, y)
4. Input.dispatchMouseEvent
   · type: 'mouseMoved'
   · type: 'mousePressed'
   · type: 'mouseReleased'
### 5.7 微信视频号微前端 iframe 与 CDP 物理点击加固

微信视频号助手前端基于微前端架构设计，发布表单、上传组件和发表按钮大多深度嵌套在独立的子 `<iframe>` 以及 Shadow DOM 中。同时微信具备极强的防爬虫和安全机制，使得该平台自动化的设计非常独特。

#### 1. 跨级嵌套 iframe 物理坐标叠加定位（getAbsoluteRect）

由于发表按钮挂载在子 `<iframe>` 作用域内，`getBoundingClientRect()` 所能提取到的像素坐标仅是相对于当前 `iframe` 视口的相对位置。如果将该相对坐标直接传给 Electron 主进程的 CDP 点击服务，会因忽略了 `iframe` 本身在 parent 文档中的偏移量导致鼠标点击发生大面积偏移。

**物理坐标解析叠加算法**：
```javascript
function getAbsoluteRect(node) {
  var rect = node.getBoundingClientRect();
  var left = rect.left;
  var top = rect.top;
  var winRef = node.ownerDocument.defaultView;
  // 向上级递归累加所有 parent iframe 节点的视口绝对偏移量，直到最外层主窗口
  while (winRef && winRef !== window) {
    try {
      var frameElement = winRef.frameElement;
      if (frameElement) {
        var fRect = frameElement.getBoundingClientRect();
        left += fRect.left;
        top += fRect.top;
      }
    } catch(e) {
      break;
    }
    winRef = winRef.parent;
  }
  return { left: left, top: top, width: rect.width, height: rect.height };
}
```

#### 2. CDP isTrusted = true 底层合成点击（Input.dispatchMouseEvent）

普通的 JS 脚本模拟点击（`click()` 等）被微信官方安全机制屏蔽，因为其派发事件的 `event.isTrusted` 字段在浏览器内核中被固化标记为了 `false`。
通过前文 `getAbsoluteRect` 获得绝对视口坐标 `(x, y)` 后，主进程直接通过 CDP 发送鼠标硬件级物理模拟动作，在内核层面强制拥有 `isTrusted === true` 凭证：
```typescript
await win.webContents.debugger.sendCommand('Input.dispatchMouseEvent', {
  type: 'mouseMoved', x, y, button: 'left', clickCount: 1, buttons: 1, modifiers: 0,
});
await sleep(80);
await win.webContents.debugger.sendCommand('Input.dispatchMouseEvent', {
  type: 'mousePressed', x, y, button: 'left', clickCount: 1, buttons: 1, modifiers: 0,
});
await sleep(120);
await win.webContents.debugger.sendCommand('Input.dispatchMouseEvent', {
  type: 'mouseReleased', x, y, button: 'left', clickCount: 1, buttons: 0, modifiers: 0,
});
```

#### 3. 转码防抢跑延时机制（防过早点击）

视频上传完毕后，微信后台会在页面端同步进行分片转码、首帧封面生成以及数据入库归档。上传刚结束时，发表按钮虽已呈亮色，但此时发送点击会因视频“未就绪/转码中”而直接被微信后台丢弃。
*   **解决方案**：系统在上传成功轮询归档后，强制设定 `await sleep(4500)` 的静置延时以等待转码就绪。内容填充后亦设定 `await sleep(2000)` 以等待 React 组件表单状态绑定和重绘校验同步，极大提升了发表通过率。

### 5.8 微信公众号：仅账号管理模式与 Cookie 登录态检测

微信公众号（mp.weixin.qq.com）由于发布流程需要管理员扫码确认，**暂不支持自动发布**，仅实现账号管理功能（登录态检测、账号信息提取、打开创作中心）。

#### 1. 平台特点与限制

| 特性 | 说明 |
|------|------|
| 发布方式 | 需要管理员扫码确认，无法纯自动化发布 |
| 登录方式 | 微信扫码登录 |
| 登录态标识 | `data_bizuin` + `bizuin` + `slave_user` 等 cookie |
| 首页跳转限制 | 直接访问 `/cgi-bin/home` 会被重定向到登录页，必须从首页进入 |
| 账号信息位置 | 左侧个人信息卡片（`.weui-desktop_name` / `.weui-desktop-account__img`） |

#### 2. Cookie 登录态检测策略

微信公众号的登录态检测采用 **cookie 为主、DOM 为辅** 的策略：

**核心登录 cookie（三者有其二即可判定登录）：**
- `data_bizuin` — 公众号业务标识（登录后必有）
- `bizuin` — 业务唯一标识（登录后必有）
- `slave_user` — 从账号用户信息（登录后必有）

**辅助登录 cookie：**
- `data_ticket` — 数据票据
- `ticket_id` — 登录票据（部分登录流程可能不存在，不作为必要条件）
- `mm_lang` — 语言设置

**强登录态判定**：当核心 cookie 数量 ≥ 3 或 ≥ 2 + `data_ticket` 时，即使页面 URL 看起来像登录页（如被重定向），也判定为已登录。这解决了"健康检测加载后台 URL 被重定向到登录页但 cookies 仍然有效"的问题。

#### 3. 账号信息提取

| 字段 | 提取方式 | 选择器 / 来源 |
|------|----------|---------------|
| 昵称 | DOM 提取 | `.weui-desktop_name`（优先）、`.weui-desktop-account__nickname` |
| 头像 | DOM 提取 | `.weui-desktop-account__img`（优先）、`img[src*="wx.qlogo.cn"]` |
| 公众号ID | Cookie 提取 | `slave_user` 字段（取 `@` 前部分） |
| 粉丝数（总用户数） | DOM 提取 | `.weui-desktop-user_num .weui-desktop-user_sum span` |
| userId | URL 提取 | URL 中的 `token=` 参数 |

#### 4. homeUrl 设计

`homeUrl` 设为 `https://mp.weixin.qq.com/`（首页），而不是后台页面。原因：
- 直接访问后台 URL（如 `/cgi-bin/home`）会被微信安全机制重定向到登录页
- 从首页进入时，微信会验证 cookies 并自动跳转到对应的后台首页
- 健康检测和打开创作中心都使用首页作为入口，确保登录态正确加载

### 5.9 React 受控组件：标题填写时机问题

如果填写标题后立即填写正文（都在 React 页面中），会遇到以下问题：

```
标题 input (受控组件):
  state: { title: '' }
  onChange: e => setState({ title: e.target.value })
  value: this.state.title         ← 渲染出来的 value 永远跟 state 同步

时序:
  t1: JS 设置 input.value = '我的标题'   ← 改了 DOM，没改 state
  t2: React 重新渲染（其他字段更新时）  ← 重新渲染 input，state 还是 ''
  t3: input.value = ''                   ← 标题被清空！
```

**解决**：用 `document.execCommand('insertText')` 代替直接赋值：

```javascript
input.focus();
document.execCommand('selectAll');
document.execCommand('delete');
document.execCommand('insertText', false, '我的标题');
// → execCommand 会触发原生 input 事件，React 监听该事件
// → 调用 onChange(e)，正确更新 state.title = '我的标题'
// → 下次重渲染保持正确值
```

### 5.8 导航跟踪器（NavigationTracker）

**问题背景**：发布流程中，文件上传后抖音/xhs 会触发 SPA 跳转到内容编辑页，此时立即 `executeJavaScript` 会失败，原因是：
- `Render frame was disposed before WebFrameMain could be accessed`
- `document.body is null`
- `context is invalid`

**解决方案**：在发布窗口创建时就挂接 `NavigationTracker`，每次执行 DOM 操作前调用 `tracker.isNavigating()` 或 `tracker.waitForStable()`。

```typescript
// 内部实现思路
lastNavigationAt: number;      // 最近一次导航事件的时间
listeners: Array<() => void>;  // 监听句柄，dispose 时清理

// 监听事件：did-navigate / did-navigate-in-page / did-frame-navigate / did-start-loading / did-stop-loading
// 每次触发 → lastNavigationAt = Date.now()

isNavigating() {
  return Date.now() - lastNavigationAt < 500 || win.webContents.isLoading();
}

waitForStable(minStableMs = 1000, timeoutMs = 10000) {
  while (Date.now() - start < timeoutMs) {
    if (!isNavigating() && Date.now() - lastNavigationAt >= minStableMs) return;
    await sleep(200);
  }
  log('warn', 'navigation', `稳定超时 ${timeoutMs}ms，继续后续流程`);
}
```

### 5.4 evalJS 重试执行器

封装 `win.webContents.executeJavaScript`，带 **最大 6 次重试**和**智能退避**：

```typescript
function isTransientEvalError(msg: string): boolean {
  return /frame was disposed|Render frame|navigation|No current|Cannot read property|
          document.body is null|null object|context is invalid|Execution context was destroyed|
          No frame|target frame|subframe/i.test(msg);
}

for (let attempt = 0; attempt < maxRetries; attempt++) {
  try {
    const result = await win.webContents.executeJavaScript(code);
    if (attempt > 0) log('debug', 'eval', `${desc} 重试成功（第${attempt+1}次）`);
    return result;
  } catch (err) {
    const msg = err.message;
    if (isTransientEvalError(msg) && attempt < maxRetries - 1) {
      const backoff = Math.min(2000, 250 * (attempt + 1));
      await sleep(backoff);
      continue;
    }
    throw err;
  }
}
```

### 5.5 React 受控组件兼容

抖音/xhs 的标题输入框都是 React `useState` 受控组件。直接 `el.value = 'xxx'` 不生效，因为 React 内部 state 没变，下次 re-render 就会被还原。

**解决方案**：

```javascript
// 1. 通过原生 setter 赋值（绕过 React 的 value setter 拦截）
const _desc = Object.getOwnPropertyDescriptor(HTMLInputElement.prototype, 'value');
if (_desc && _desc.set) _desc.set.call(el, value);

// 2. 派发 React 监听的原生事件
el.dispatchEvent(new Event('input', { bubbles: true }));
el.dispatchEvent(new Event('change', { bubbles: true }));
el.dispatchEvent(new Event('blur', { bubbles: true }));
```

`buildFillTitle` / `buildFillContent` 都使用此模式，并会按评分排序候选元素：
- **标题**：关键词 `标题/title/作品标题` + 输入框 tag + 高度评分
- **正文**：关键词 `描述/正文/内容/简介/description` + textarea/contenteditable + 排除"标题"字段
- **话题**：用户输入的 `tag1, tag2` 自动解析为 `#tag1 #tag2` 行文本，附加在内容末尾

### 5.6 文件上传（CDP setFileInputFiles）

Electron 默认限制：从渲染进程的 JS 无法直接设置 `<input type=file>` 的 `files`（浏览器安全策略）。

**解决方案**：使用 Chrome DevTools Protocol 的 `DOM.setFileInputFiles` 命令：

1. 通过 `debugger` 协议附加到发布窗口的 webContents
2. `DOM.getDocument` 获取整棵 DOM 树
3. 查找第一个 `<input type="file">` 的 `nodeId`
4. `DOM.setFileInputFiles({ nodeId, files: [path1, path2, ...] })` 注入本地文件路径
5. 在页面上模拟 change 事件（让 React 感知到上传了文件）
6. 断开 debugger 连接

这种方式是自动化场景的标准做法，Selenium/Puppeteer 内部也是同样机制。

## 六、反崩溃策略（重点：抖音发布场景）

### 6.1 问题背景

在 Electron 28 上运行抖音发布流程时，文件上传完成后约 2~5 秒内会触发渲染进程崩溃：

```
[ERROR:CONSOLE(1)] exitCode: -1073741819 (STATUS_ACCESS_VIOLATION)
"Render frame was disposed before WebFrameMain could be accessed"
```

**根因**：抖音创作者中心在视频上传后会：
1. 启动 `WebAssembly` 模块进行人脸检测/封面推荐（`bytenn.wasm`）
2. 在 `WebWorker` 中做视频帧提取和解码
3. 使用 `WebGL` 渲染预览画面
4. 尝试硬件加速视频解码（GPU 路径）

这几条路径在 Electron 28 + Windows 组合下（尤其是无沙盒环境）都不稳定，容易触发 `STATUS_ACCESS_VIOLATION`。

### 6.2 防御层次（最终方案：Electron 31）

| 层次 | 位置 | 说明 |
|:---|:---|:---|
| 1. **Electron 版本升级** | `package.json` | 从 28 → 31.7.7（Chromium 126）。Chromium 官方修复了大量无沙盒环境下 GPU/Worker 相关的稳定性问题 |
| 2. **全局 GPU 禁用** | `src/main/index.ts` | `app.disableHardwareAcceleration()` 全局关闭硬件加速 |
| 3. **Chromium 命令行开关** | `src/main/index.ts` | `disable-gpu-sandbox` / `disable-gpu-compositing` / `disable-accelerated-video-decode` / `disable-accelerated-video-encode` / `disable-webgl` |
| 4. **软件 GL 回退** | `src/main/index.ts` | `use-gl=swiftshader`，GPU 禁用后使用软件渲染 |
| 5. **V8 堆内存** | `src/main/index.ts` | `js-flags=--max-old-space-size=8192`，防止创作中心大页面 OOM |
| 6. **崩溃自动恢复** | `platforms/shared.ts` | `render-process-gone` 事件 → `win.reload()` 自动 reload 1 次（兜底） |

### 6.3 为什么升级到 31 是最终方案

- Chromium 112 (Electron 28) → 126 (Electron 31) 之间跨越 14 个主要版本
- 官方 Release Notes 中 GPU、WebAssembly、Worker 相关的修复有数十项
- 在相同测试场景下：Electron 28 崩溃率 ≈ 80%，Electron 31 崩溃率 ≈ 0%
- 升级后可以去掉大量激进的 polyfill 代码（如冻结 WebAssembly/Worker），减少对页面的干扰，让视频上传的 WebAssembly 流程正常工作

### 6.4 注意事项

- `app.disableHardwareAcceleration()` **必须在 app ready 前调用**，否则 Chromium 已经启动 GPU 进程后无法回退
- 发布速度会变慢（软件渲染），但整体稳定性大幅提升
- 对正常使用（用户手动在浏览器窗口操作）完全无影响

## 八、安全设计

### 8.1 Sandbox + Context Isolation

```typescript
// 所有 BrowserWindow（主窗口 + 发布窗口 + 授权窗口）
new BrowserWindow({
  webPreferences: {
    sandbox: true,
    contextIsolation: true,
    nodeIntegration: false,
    nodeIntegrationInWorker: false,
    partition: `persist:account_${id}`, // 额外的账号隔离
  },
});
```

即使主窗口页面被攻破，也无法：
- 访问 Node.js API
- 访问其他账号的 cookies（partition 隔离）
- 向外部 URL 发送 IPC 消息（主进程 `notifyStatus` 对外部 URL 窗口做了过滤）

### 8.2 Preload 白名单 API

```typescript
// preload/index.ts 暴露的 API
window.electronApi = {

### 8.3 Cookie 加密存储

账号凭证包含平台 session cookie，属于高度敏感数据。

```typescript
// SecureStore.ts 封装
import Store from 'electron-store';
import { safeStorage } from 'electron';

const store = new Store({
  name: 'flowx-secure',
  encryptionKey: 'flowx-desktop-v1',  // 加密文件名（防止本地明文扫描）
});

// 保存：先通过 safeStorage.encryptString 加密，再写入
export function saveCredentials(acc: AccountCredential) {
  if (safeStorage.isEncryptionAvailable()) {
    acc.cookies = acc.cookies.map(c => ({
      ...c,
      value: safeStorage.encryptString(c.value).toString('base64'),
    }));
  }
  store.set(`account.${acc.id}`, acc);
}
```

**加密级别**：
- Windows：DPAPI（Data Protection API），绑定当前 Windows 用户
- macOS：Keychain
- Linux：libsecret

攻击者即使拿到 `flowx-secure.json` 文件，在不同机器/不同用户账号下也无法解密。

### 8.4 IPC 推送的窗口过滤（关键安全点）

`PublishEngine.notifyStatus` 的实现细节：

```typescript
// 向主窗口推送状态，但跳过加载外部 URL 的窗口（如抖音创作中心）
for (const win of BrowserWindow.getAllWindows()) {
  const wc = win.webContents;
  const url = wc.getURL();
  if (url.startsWith('http://') || url.startsWith('https://')) continue;  // 跳过外部页
  if (wc.hostWebContents) continue;                                        // 跳过 iframe 宿主
  wc.send('publish:statusChanged', payload);
}
```

**为什么需要过滤**：
- 当主窗口是 `file://dist/index.html`，发布窗口是 `https://creator.douyin.com/...`
- 把 IPC 消息发到 `creator.douyin.com` 窗口，该页面没有我们的 preload，没有监听 `publish:statusChanged`
- 但 Chromium 的 WebFrameMain 在收到 `send` 时仍会尝试执行框架内的 JS
- 在某些时序下（页面恰好正在跳转）会触发 `Render frame was disposed`

这是一条反复踩坑后才稳定下来的关键防御逻辑。

## 九、发布流程中的状态同步机制

### 9.1 三条状态同步途径（互补而非替代）

```
途径 1: 前端登记
  Publish.vue submit()
    └─> publishStore.liveTasks[taskId] = { status: queued, progress: 0 }
  [目的：立即显示 UI，避免"提交后没反应"]

途径 2: 主进程主动推送
  PublishEngine.notifyStatus(task, verbose=true)
    └─> wc.send('publish:statusChanged', payload)
          └─> publishStore._applyStatusUpdate(raw)
                └─> Vue 响应式更新 → UI 刷新
  [目的：实时同步正在发生的进度]

途径 3: 前端主动拉取
  publishStore.submit() 末尾
    └─> electronApi.getPublishProgress(taskId)
          └─> 合并到 liveTasks
  [目的：防止第一次推送被遗漏时 UI 状态不一致]
```

### 9.2 任务完成后的自动清理

用户痛点：任务成功后"任务进行中"面板还一直显示着已完成的任务，页面显得混乱。

解决方案（`stores/publish.ts _applyStatusUpdate`）：

```typescript
// success → 8 秒后从 liveTasks 移除
// failed → 15 秒后移除（让用户有时间查看错误信息）
// running → 重新进入时清除之前的移除计划（防止从 failed 重试时被误删）

if (newStatus === 'success' || newStatus === 'failed') {
  const delay = newStatus === 'success' ? 8000 : 15000;
  if (this._autoRemoveTimers[taskId]) clearTimeout(this._autoRemoveTimers[taskId]);
  this._autoRemoveTimers[taskId] = setTimeout(() => {
    delete this.liveTasks[taskId];
    delete this._autoRemoveTimers[taskId];
  }, delay) as unknown as number;
} else if (this._autoRemoveTimers[taskId]) {
  clearTimeout(this._autoRemoveTimers[taskId]);
  delete this._autoRemoveTimers[taskId];
}
```

## 十一、扩展指南

### 11.1 新增一个平台

只需在 `src/main/services/platforms/` 新增一个文件，文件内调用 `registerPlatform()` 即可被系统发现：

```typescript
// Step 1: 新增 platforms/example-platform.ts
import { runStandardPublish, evalJS } from './shared';
import { registerPlatform } from './registry';

const meta = {
  key: 'example',
  name: '示例平台',
  icon: '📝',
  publishUrl: 'https://creator.example.com/publish',
  homeUrl: 'https://creator.example.com/home',
  contentTypes: ['video', 'image'],
  capabilities: { publishVideo: true, publishImage: true, publishArticle: false },
  loginKeywords: ['我的内容', '发布作品'],
  // ...其他元信息
};

function detectLoggedIn(win) {
  return evalJS(win, `
    document.body.innerText.includes('我的内容') ||
    document.body.innerText.includes('发布作品')
  `, 'detect-login', log);
}

function publishVideo(accountId, request, onProgress) {
  return runStandardPublish(accountId, request, onProgress, {
    platform: 'example',
    meta,
    detectLoggedIn,
    publishKeywords: ['发布', '确认发布', '立即发布'],
    enableConfirmStep: true,
    enablePostClickVerify: true,
    fillWaitMs: 2000,
  });
}

// 统一注册（index.ts 会自动 import 本文件，注册在 registry 中）
registerPlatform({
  key: 'example',
  meta,
  capabilities: meta.capabilities,
  detectLoggedIn,
  publishVideo,
  publishImage: publishVideo,
});

export default { key: 'example', meta, capabilities: meta.capabilities, detectLoggedIn, publishVideo, publishImage: publishVideo };
```

**无需修改 PublishEngine / PlatformDispatcher** —— PlatformDispatcher 自动通过 `getPlatformAdapter()` 发现新平台。

### 11.2 新增一个内容类型（如"直播"）

1. 在 `types/index.ts` 的 `ContentType` 添加 `'live'`
2. 在平台 `meta.capabilities` 添加 `publishLive: true`
3. 在平台适配器实现 `publishLive()` 方法
4. 在 `PlatformDispatcher.ts` 的 switch 中添加 `case 'live'` 分支
5. 前端 Publish.vue 自动会根据账号 `capabilities.publishLive` 显示对应的账号卡片

### 11.3 调试技巧清单

| 问题 | 排查方法 |
|:---|:---|
| 发布成功但 UI 没显示 | 打开 Publish.vue → 切换 `showDebug` → 查看 `_applyStatusUpdate` 日志，检查是否收到了 IPC 推送 |

### 13.1 图文发布的价值

与视频发布的核心区别

图文发布（image content publishing），是 v1.1 新增的核心能力。与视频发布相比，主要差异如下：

| 维度 | 视频发布 | 图文发布 |
|:---|:---|:---|
| 上传内容 | `.mp4` / `.mov` 等视频文件 | `.jpg` / `.png` / `.webp` 等图片文件 |
| 上传 input | `accept="video/*"` | `accept="image/*"` |
| 富文本编辑 | 简单 contenteditable（描述信息 | ProseMirror / Tiptap 富文本编辑器 |

| 字数限制 | 快手 500 / 抖音 1000+ | 小红书标题 20 / 正文 1000，快手描述 500 |

| URL 入口 | 平台默认发布页 | 需指定特殊参数切换到图文模式 |

### 13.3 uploadViaCDP 的 contentType 参数

`uploadViaCDP` 函数签名：

```typescript
export async function uploadViaCDP(
  win: BrowserWindow,
  files: string[],
  log: LogFn,
  contentType?: 'video' | 'image',  // ← 关键参数
): Promise<boolean>
```

内部逻辑：

1. CDP `DOM.getDocument({ depth: -1 }) 遍历 DOM 树
2. 收集所有 `input[type="file"]` 元素
3. **按 contentType 排序**：
   - `contentType='image'` 优先匹配 `accept` 属性含 `image/` 的 input
   - `contentType='video'` 优先匹配 `accept` 属性含 `video/` 的 input
   - 无 accept 或 accept='*/*' 的作为兜底
4. 跳过 `aria-hidden="true"` 容器内的隐藏 input
5. `DOM.setFileInputFiles(nodeId, files)` 注入文件
6. 手动派发 `change` + `input` 事件（让 React 感知到文件变化
7. 等待 `waitForUploadComplete` 轮询检测上传状态

**关键修复（小红书 ProseMirror 富文本编辑器

小红书的正文编辑区是 ProseMirror/Tiptap 富文本编辑器。

**为什么直接 `innerText` 不生效**：

```javascript
// ❌ 失败方案：
editable.innerText = content;
// ProseMirror 在下次 keydown 时还原 state，内容丢失

// ✅ 正确方案（在 xiaohongshu.ts buildFillContentScript 内实现
editable.focus();
document.execCommand('selectAll');
document.execCommand('delete');
// 逐行 insertText
document.execCommand('insertText', false, line + '\n');
// → ProseMirror 监听 input 事件
// → 更新内部 state.doc 正确更新
```

**为什么逐行插入**：ProseMirror 的 `insertText` 命令一次只能处理一行文本。多行内容需要分多次 execCommand 调用。

### 13.5 小红书发布按钮的 closed shadow DOM

小红书发布按钮的 DOM 结构：

```html
<xhs-publish-btn submit-text="发布笔记">
  <!-- #shadow-root (closed)  ← 外部 JS 无法访问
    <div class="publish-page-publish-btn">
      <button class="ce-btn bg-red">发布笔记</button>
    </div>
  </xhs-publish-btn>

**三层策略**：

1. **第一层：尝试 `element.click()` — 轻量级尝试点击 host 元素（有时平台会在 host 上也绑定事件
2. **第二层：CDP 穿透点击 — 使用 `DOM.getDocument({ pierce: true })` 强制穿透 shadow DOM
3. **第三层：坐标点击 — 对内部 button 的 `getBoxModel` 后，`Input.dispatchMouseEvent` 合成鼠标事件

```typescript
// cdpClickPublishButton 内部逻辑
1. CDP DOM.getDocument({ depth: -1, pierce: true })  ← 关键参数
2. 遍历所有候选元素的 nodeId
3. DOM.getBoxModel(nodeId) → content[0..3] / margin/padding/border
4. 计算中心坐标 (centerX, centerY)
5. Input.dispatchMouseEvent
   · type: 'mouseMoved' → 'mousePressed' → 'mouseReleased'
   · button: 'left'
   · clickCount: 1
6. 合成鼠标事件不受 shadow DOM 隔离，真正的按钮可接收到鼠标事件
7. 即使 JS click() 不生效时也能点击

### 13.6 字数与标签限制的统一约束

各平台的图文发布有严格的字数/标签限制，超过限制会导致发布失败。

| 平台 | 标题字数 | 正文字数 | 标签数 |
|:---|:---|:---|:---|
| 小红书 | **最多 20** | 最多 1000 | 话题最多 10 个 |
| 抖音 | 最多 80 | 最多 1000 | 最多 10 个话题 |
| 快手 | 无独立标题 | 最多 500 | 最多 **4** 个标签 |

**实现方式**：在每个平台的 `publishImage()` 内部，调用自己的 truncate 函数，而非 shared.ts 统一处理。这是平台差异隔离原则的体现。

```typescript
// 示例：小红书标题截断
const TITLE_LIMIT = 20;
const CONTENT_LIMIT = 1000;
const MAX_TAGS = 10;

function truncateText(text: string, limit: number): { text: string; truncated: boolean {
  if (!text) return { text: '', truncated: false };
  if (text.length <= limit) return { text, truncated: false };
  return { text: text.slice(0, limit - 1).trimEnd() + '\u2026', truncated: true };
}
```

超过限制时，会记录 warning 级别的日志，但不抛异常，继续后续流程。

| 发布按钮路径 | 标准 DOM button | shadow DOM（小红书） | 小红书：3 步点击 → shadow DOM button<br/>抖音：标准 DOM button |

**核心区别**：小红书文章发布不是"填入 → 点击发布"的两步流程，而是**四步流程**——进入编辑器 → 填写 → 排版 → 下一步 → 发布。这是与图文发布最大的差异，需要独立的多步点击逻辑。

### 14.2 三级分发架构（扩展 contentType）

PlatformDispatcher 从之前的 2 级（video / image）扩展为 3 级（video / image / article）：

```typescript

  platform: string,
  contentType: 'video' | 'image' | 'article',  // ← 新增 'article'
): PublishExecutor

interface PlatformAdapter {
  publishVideo(accountId, request, onProgress)      // 视频发布

  detectLogin(accountId)

}

interface PlatformMeta {
  contentLimits: { title?: number; content?: number }  // 图文/视频限制

- 抖音/小红书：`publishArticle` 已实现，返回 `PublishItemProgress`

### 14.5 文章摘要与话题处理（两大平台差异）

文章发布相比图文/视频，新增了两个平台差异很大的特性：**独立摘要字段**和**专用话题添加方式**。

#### 14.5.1 文章摘要字段

| 平台 | 摘要位置 | 字数限制 | 编辑器类型 |
|:---|:---|:---|:---|
| 抖音 | 发布页底部独立"摘要"输入框 | 30 字 | contenteditable div |
| 小红书 | 第三步（设置页）"正文描述"框 | 1000 字 | ProseMirror 编辑器（第二个） |

**小红书摘要识别的特殊处理**：

小红书文章发布有两个 ProseMirror 编辑器——第一个是正文编辑器，第二个是摘要（正文描述）编辑器。需要通过编辑器顺序而非 class 名来区分：

```typescript
// 构建填写摘要的脚本：找到第二个 ProseMirror 编辑器
function buildFillArticleSummaryScript(summary: string): string {
  return `
    (function() {
      const editors = document.querySelectorAll('.ProseMirror[contenteditable="true"]');
      if (editors.length < 2) return { success: false, error: '未找到摘要编辑器' };
      
      const summaryEditor = editors[1]; // 第二个 = 摘要编辑器
      // ... 聚焦 + 清空 + insertText 输入
      return { success: true };
    })();
  `;
}
```

**为什么不能用 DOM 直接修改**：ProseMirror 编辑器维护内部状态，直接修改 `textContent` 或 `innerHTML` 会导致：
1. 编辑器内部 state 与 DOM 不同步
2. 输入的内容在失焦后被清空
3. 无法触发平台的字数统计和校验

**正确做法**：通过 `document.execCommand('insertText')` 模拟键盘输入，触发编辑器事务更新。

#### 14.5.2 文章话题添加方式对比

| 发布类型 | 话题添加方式 | 最大数量 | 实现方式 |
|:---|:---|:---|:---|
| 视频/图文 | 正文末尾追加 `#话题` | 10 个 | JS 直接插入文本 |
| 抖音文章 | **专用弹窗搜索添加** | 5 个 | 打开弹窗 → 输入搜索 → 点击推荐项 |
| 小红书文章 | **摘要框内插入话题标签** | 未明确限制 | 摘要框中模拟键盘输入 `#话题` |

**抖音文章话题弹窗流程**：

```
点击"添加标签"按钮
    ↓
弹窗出现（包含搜索输入框 + 推荐话题列表）
    ↓
在搜索框输入关键词（不带 # 号）
    ↓
等待推荐列表加载（1-2 秒）
    ↓
点击第一个推荐项（高亮选中）
    ↓

    ↓

```

**关键注意点**：
- 搜索关键词**不带 `#` 号**，直接输入纯文字
- 最多 **5 个**话题（图文/视频是 10 个）
- 必须点击推荐列表中的项才算添加成功，直接输入不生效
- 不影响图文/视频发布的话题逻辑，完全独立代码路径

**小红书文章话题插入**：

在第三步的摘要（正文描述）框中，通过模拟键盘输入 `#话题名 ` 的方式插入话题标签：

```typescript
// 在摘要编辑器末尾追加话题
async function insertTagsInSummary(win, tags: string[]) {
  const tagText = tags.map(t => `#${t} `).join('');
  await evalJS(win, `
    const editors = document.querySelectorAll('.ProseMirror[contenteditable="true"]');
    const summaryEditor = editors[1];
    summaryEditor.focus();
    // 移到末尾
    document.execCommand('selectAll', false, null);
    document.execCommand('insertText', false, '原有内容' + '${tagText}');
  `);
}
```

> **注意**：小红书文章话题必须通过模拟键盘输入（`execCommand('insertText')`），直接修改 DOM 不会触发 ProseMirror 的事务更新，话题标签不会被平台识别。

### 14.6 代码隔离原则

文章发布的代码完全不影响图文/视频发布：

| 模块 | 代码隔离方式 |
|:---|:---|
| **`publishArticle` 方法** | 每个平台文件中独立函数体，不调用 `publishImage` 或 `publishVideo` 的任何逻辑 |
| **限制常量** | 文章有独立的 `ARTICLE_*_LIMIT` 常量，不与 `TITLE_LIMIT` / `CONTENT_LIMIT` 共用 |
| **发布结果检测** | `buildPublishResultProbeScript` 新增 `isArticleSuccess` 分支，不修改原有逻辑 |
| **按钮点击逻辑** | 小红书文章发布完全走独立的三步点击函数，不调用图文发布的 `click-publish` 代码 |
| **meta 配置** | `articleLimits` 与 `contentLimits` 是两个独立字段，前端 `Publish.vue` 按 contentType 选择读取 |
| **前端 UI** | 发布页面的「视频 / 图文 / 文章」三个 Tab 独立渲染，文章 Tab 下快手账号自动过滤 |

> **修改文章发布的任何逻辑都不影响图文/视频发布**。反之亦然。

### 14.8 Markdown 编辑器（v1.3 新增）

#### 14.8.1 功能概述

文章发布模式下新增「纯文本 / Markdown」模式切换：
- **纯文本模式**：保持原有 textarea 编辑方式
- **Markdown 模式**：自研轻量 Markdown 编辑器，左右分栏（左侧编辑 / 右侧实时预览），发布时生成 `.md` 文件上传到平台

**支持的平台**：小红书文章、抖音文章

**支持的 Markdown 语法**（与平台能力对齐，编辑器已做精简）：

| 功能 | 语法 | 快捷键 |
|------|------|--------|
| 标题 H1/H2/H3 | `#` / `##` / `###` | - |
| 加粗 | `**文本**` | Ctrl+B |
| 斜体 | `*文本*` | Ctrl+I |
| 高亮 | `==文本==` | Ctrl+H |
| 引用 | `> ` | - |
| 无序列表 | `- ` | - |
| 有序列表 | `1. ` | - |
| 图片 | `![alt](url)` | - |

> 不支持的语法（删除线、代码块、链接、分割线、任务列表等）在工具栏中已屏蔽，避免用户输入后平台无法渲染。

#### 14.8.2 前端实现

**核心组件**：`src/renderer/components/MarkdownEditor.vue`

- 自研轻量编辑器（未引入 Jodit 等重型依赖）
- 左右分栏布局，可拖拽分隔条调整宽度
- 同步滚动（编辑区滚动 → 预览区同步）
- 字数统计：排除 Markdown 标记和换行符，与平台限制计算一致
- 渲染方案：`marked` 库 + 自定义高亮语法正则（`==文本==` → `<mark>`）

**模式切换逻辑**（`PublishForm.vue`）：
- 纯文本 → Markdown：将当前纯文本内容直接作为 Markdown 源码迁移
- Markdown → 纯文本：清洗 Markdown 标记后回填，保证内容不丢失
- 草稿保存/加载完整支持 Markdown 模式

#### 14.8.3 发布流程

**小红书**：
1. 进入一键排版页面
2. 点击工具栏文档导入图标 → 弹出导入弹窗
3. 点击上传区域（`.upload-area`）→ 触发 file input 动态生成
4. CDP `DOM.setFileInputFiles` 注入 .md 文件
5. 等待内容自动填充 → 关闭弹窗 → 后续流程与纯文本一致

**抖音**：
1. 检测引导页（只检测不自动点击）
2. Markdown 模式：点击「一键导入」按钮
3. 等待上传弹窗 → CDP 注入 .md 文件
4. 页面自动跳转编辑器并填充内容 → 后续流程与纯文本一致

**失败回退机制**：任何上传步骤失败时，自动清洗 Markdown 标记为纯文本，通过常规 CDP 输入方式填写正文，保证发布成功率。

#### 14.8.4 关键技术点

- **uploadFileToInput**：`shared.ts` 中通用 CDP 文件上传工具，支持轮询查找动态生成的 file input

- **字数计算一致性**：前端编辑器、表单验证、平台适配器三处使用相同的 Markdown → 纯文本清洗规则

### 14.9 知乎视频发布（v1.2 新增平台）

#### 14.9.1 概述

知乎是 v1.2 阶段新增的视频发布平台，实现文件为 `src/main/services/platforms/zhihu.ts`。发布页面为 `https://www.zhihu.com/upload-video`。

知乎视频发布的特点：
- 使用 **Draft.js** 作为描述富文本编辑器（区别于小红书的 ProseMirror、抖音的 contenteditable）
- **视频标记**为必填项，通过 Modal 弹窗选择（动态渲染，非普通下拉框）
- 标题使用 React 受控组件（textarea）
- 封面图由平台自动从视频中提取，用户可逐帧选择

#### 14.9.2 发布流程（10 步）

```

步骤 2: 加载 https://www.zhihu.com/upload-video
步骤 3: 导航跟踪器等待页面稳定
步骤 4: 登录态检测（z_c0 cookie + DOM 元素双重验证）
步骤 5: CDP 注入视频文件到 input.VideoUploadButton-fileInput
步骤 6: 等待上传完成（表单 + 封面 + 标题三重探针，最长5分钟）
步骤 7: 填写标题（50字限制，nativeSetter + input 事件）
步骤 7.5: 选择视频标记【必填】（打开Modal → 等待渲染 → 选选项 → 确认）
步骤 8: 填写描述（1000字限制，Draft.js + execCommand insertText）
步骤 9: 点击发布按钮（.VideoUploadForm-submitButton）

#### 14.9.3 核心技术难点

**难点 1：视频标记 Modal 的动态渲染**

- 视频标记是知乎视频发布的必填项（共 6 个选项：内容无需标注、含 AI 生成内容、含虚构演绎内容、内容含营销信息、内容为转载、个人观点仅供参考）
- **问题**：点击触发区域后，Modal 才动态创建并挂载到 DOM，同步脚本无法一步完成"点击 → 选择 → 确认"
- **解决方案**：拆分为三步异步流程，外层控制时序
  1. `buildOpenVideoTagModalScript()` — 点击触发按钮打开 Modal
  2. 外层轮询等待 Modal 出现（每 300ms 检查，最多 3 秒）
  3. `buildSelectAndConfirmVideoTagScript()` — 在已渲染的 Modal 中选择选项并点击确认
- 触发方式（按优先级）：`.VideoUploadForm-videoTypeSelectOverlay` → `.VideoUploadForm-select [role="combobox"]` → 文本匹配
- 最多重试 3 次，每次失败自动关闭 Modal 后重试

**难点 2：Draft.js 富文本编辑器填写**

- 知乎描述使用 Draft.js 框架，直接修改 DOM 不生效（框架有内部 EditorState）
- **解决方案**：使用 `document.execCommand('insertText')` 模拟原生输入
- 触发完整的编辑事件管线，Draft.js 能正确捕获并更新内部 state
- 操作步骤：聚焦 → selectAll → delete → insertText → 触发 input/keyup 事件

**难点 3：React 受控组件标题填写**

- 标题 textarea 是 React 受控组件，直接设置 `value` 属性不生效
- **解决方案**：`Object.getOwnPropertyDescriptor` 获取原生 setter → 调用后触发 `input/change` 事件
- 与抖音图文/文章发布方案一致

**难点 4：上传完成判定**

- 视频上传后需经历上传、转码、封面生成等多个后台步骤
- **三重验证探针**（全部满足才算完成）：
  1. 表单容器 `.VideoUploadForm` 存在
  2. 封面图 `img.VideoUploadForm-image` 有 src
  3. 标题输入框有自动填充的文件名

|:---|:---|:---|

| 描述 | 1000 个字符 | `.public-DraftEditor-content`（Draft.js） |

### 14.10 知乎图文发布（v1.2 扩展）

#### 14.10.1 概述

知乎图文发布基于知乎的**"发想法"**功能实现，发布页面为 `https://www.zhihu.com/creator`（创作者中心）。与其他平台不同，知乎没有专门的图文发布入口，而是通过"分享此刻的想法"功能发布图文内容。

知乎图文发布的特点：
- 使用 **Draft.js** 作为正文富文本编辑器（与知乎视频描述同一技术栈）
- **图片独立于编辑器**：图片上传后显示在 `ImageArea` / `DraggableTags` 区域，而非嵌入 Draft.js 编辑器内部
- **编辑器折叠/展开两态**：初始状态为折叠的提示文本（`css-1lkz3hi`），点击后才展开完整表单
- 话题（标签）通过 `#话题名#` 格式直接写在正文中
- 标题使用 React 受控组件（textarea）

#### 14.10.2 发布流程（10 步）

```
步骤 1: 创建独立窗口（账号隔离 partition）
步骤 2: 加载 https://www.zhihu.com/creator
步骤 3: 导航跟踪器等待页面稳定
步骤 4: 登录态检测（z_c0 cookie + DOM 元素双重验证）
步骤 5: 激活编辑器（点击 css-1lkz3hi 提示文本，展开完整表单）
步骤 6: 填写标题（50字限制，nativeSetter + input 事件）
步骤 7: 上传图片（点击图片按钮 → Modal弹窗 → CDP注入 → 等待上传完成 → 点击插入）
步骤 8: 填写正文 + 话题（1000字限制，Draft.js + paste事件/CDP键盘输入）
步骤 9: 点击发布按钮（WritePinToolbar 右侧发布按钮）
步骤 10: 等待发布结果（URL变化 + 成功文本轮询）
```

#### 14.10.3 核心技术难点

**难点 1：编辑器激活状态判断**

- 知乎想法编辑器有折叠/展开两种状态，DOM 元素始终存在但视觉上折叠
- **问题**：仅检测 DOM 元素存在性会误判为"已激活"，跳过点击步骤
- **解决方案**：通过 `css-1lkz3hi` 提示文本的**可见性**判断状态
  - `css-1lkz3hi` 可见 → 编辑器未激活 → 需要点击激活
  - `css-1lkz3hi` 不可见 + `ImageArea` 可见 → 编辑器已激活
- 多层可见性检测：`getComputedStyle(display/visibility)` + `offsetParent`

**难点 2：图片上传弹窗与 Modal 交互**

- 点击图片按钮后弹出上传 Modal，文件输入在 Modal 内部
- **问题**：CDP 注入需要精确的 `nodeId`，且 Modal 内元素动态渲染
- **解决方案**：双方案获取 nodeId
  - 方案1：JS 定位 + `DOM.querySelector`（先给 file input 加标记，再用 CDP 查询）
  - 方案2：`DOM.getDocument(depth:-1, pierce:true)` 全量遍历兜底
- 上传状态检测：通过按钮 class 变化判断
  - `css-12pk5sz` → 上传中（按钮不可点击）
  - `css-owamhi` → 上传完成（按钮可点击）
- 遮罩层保护：注入脚本阻止遮罩层点击事件冒泡，防止误关弹窗

**难点 3：图片区域识别（非编辑器内图片）**

- 知乎图文的图片不嵌入 Draft.js 编辑器，而是在独立的 `ImageArea` 区域
- **问题**：在编辑器内查找图片永远找不到，导致误判上传失败
- **解决方案**：优先检测 `ImageArea` / `DraggableTags` 区域的图片
  - 统计 `DraggableTags-tag` 数量（排除 plus 按钮）
  - 验证图片 src 为真实 URL（排除 ZDI 图标等 SVG）
  - 测试模式检测逻辑同步更新

**难点 4：Draft.js 正文填写（React 状态同步）**

- 知乎正文使用 Draft.js 编辑器，简单 DOM 操作不会更新 React state
- **问题**：内容写入了 DOM，但页面视觉上不显示，需用户点击后才回显
- **解决方案**：四层兜底方案
  - 方案1：`ClipboardEvent('paste')` — 粘贴事件，Draft.js 原生支持
  - 方案2：逐个字符触发 `beforeinput + input` 事件序列
  - 方案3：`document.execCommand('insertText')`
  - 方案4：CDP `Input.dispatchKeyEvent` 模拟真实键盘输入（最可靠）
- 增加 `debug` 字段，记录每种方案的执行结果

#### 14.10.4 字数与内容限制

## 十五、指纹隔离、代理配置、定时发布与多分类管理 (V4.0 重磅演进)

### 15.1 浏览器环境指纹与代理隔离设计
- **服务载体 (BrowserEnvService)**：封装独立服务类统一管理代理 IP (http/socks5) 与环境配置 (User-Agent)。
- **隔离应用 (applyEnvironment)**：
  - 基于 partition 级 Session 对象进行设置。
  - 调用 `session.setUserAgent` 设置特定环境的浏览器标识。
  - 调用 `session.setProxy` 定向配置代理出网，隔离物理网络。
  - 对代理身份鉴权进行 `sess.on('login')` 的自动拦截填充，每次配置前清空所有监听以防范内存泄露。
- **作用域链**：
  - 新增授权（BrowserWindow）
  - 批量健康检测（隐藏模式静默检测）
  - 账号后台刷新（自动载入）
  - 访问创作者中心（主进程跳转）
  - 自动发布流程（makePublishWindow 统一窗口创建）
  在以上场景所创建的所有 `BrowserWindow` 会话中强制注入账号关联的隔离环境，杜绝主机真实网络与指纹泄露。

### 15.2 一键发布定时调度设计
- **内存任务队列 (scheduledTimers)**：主进程 `PublishEngine` 内部维护 `Map<string, NodeJS.Timeout>`，对置为定时发送的任务做毫秒级延时发布。
- **关机恢复逻辑**：应用再次启动时读取磁盘任务：
  - 过去未执行任务：变更为 `failed`（“在应用关闭期间错过了预定的发布时间”）。
  - 未来待执行任务：依据当前时刻重新计算相对延时，一键注入 `setTimeout` 恢复后台运行。
- **随时取消支持**：可点击“取消发布”手动清空定时器句柄，任务状态同步为 `cancelled`。

### 15.3 账号多分类绑定与筛选
- 账号属性 `categoryIds?: string[]` 支持最多绑定 5 个分类标签。
- 一键发布界面追加分类下拉框筛选，多账号全选和布局进行分类联动过滤。

### 16.3 核心服务

#### 16.3.1 BrowserService（浏览器视图管理）

核心职责：管理 WebContentsView 的创建、销毁、布局，以及标签页状态维护。

关键设计：
- **多标签管理**：`Map<string, TabState>` 维护每个标签的 URL、标题、加载状态、WebContentsView 实例
- **视图切换**：切换标签时通过 `setBounds` 隐藏非活动标签，避免每次重建 WebContentsView 导致重载
- **布局同步**：监听窗口 resize 和分栏拖拽，实时调用 `setBounds` 调整浏览器视图位置和尺寸
- **环境应用**：每个标签创建时根据选择的环境 ID 应用 User-Agent 和代理配置
- **事件转发**：将 WebContents 的加载进度、标题更新、SSL 错误等事件转发到渲染层

#### 16.3.2 ContentExtractor（内容提取引擎）

三级提取策略，按优先级依次尝试：

```
┌──────────────────────────────────────────────────────────┐
│                   extractContent()                        │
│                                                          │
│  第一级：站点规则适配（confidence: 85-100）              │
│    ├─ 匹配站点规则 → 精准 CSS 选择器提取                 │
│    └─ 已适配：微信/知乎/头条/36氪/简书/少数派/CSDN/掘金  │
│                                                          │
│  第二级：Readability.js（confidence: 50-100）            │
│    ├─ 注入页面上下文执行，基于文本密度算法提取正文        │
│    └─ 输出标准化的 title/content/excerpt/images          │
│                                                          │
│  第三级：文本密度算法（confidence: 40-95）               │
│    ├─ 遍历 DOM，计算各元素文本密度                       │
│    └─ 选择密度最高的容器作为正文区域                     │
│                                                          │
│  结果：返回 ExtractedContent（含置信度和策略标识）       │
└──────────────────────────────────────────────────────────┘
```

关键特性：
- **文本清理管线**：七步清理（零宽字符移除 → 换行符统一 → 空格归一 → 行首尾去空格 → 连续空行合并 → 首尾去空行 → 多余空格合并）
- **图片五级过滤**：广告域名过滤 → data URI 过滤 → 尺寸过滤（≥50px）→ 比例过滤（0.3-5）→ 内容图片判定（≥200x100）
- **图片元数据增强**：宽度/高度/宽高比/alt/位置索引/isLikelyContent 标记

#### 16.3.3 BrowserHistoryService（书签与历史记录）

- **书签**：支持文件夹分组、拖拽排序、导入导出
- **历史记录**：按天分组、搜索过滤、一键清除
- **持久化**：SecureStore 存储，按用户隔离

- **点击确认**：点击元素后，提取该元素内容并退出选择模式
- **ESC 取消**：按 Escape 键退出选择模式，不提取
- **方向键↑↓**：向上/向下切换 DOM 层级（父元素/子元素），方便精确选择
- **视觉反馈**：浏览器区域显示橙色边框和浮动提示条

选择结果通过 `console-message` 事件传递（`__FLOX_SELECT__:` 前缀），主进程解析后调用提取函数。

#### 16.4.3 光标位置插入

手动提取与自动提取的填充策略不同：

| 提取方式 | 填充模式 | 说明 |
|:---|:---|:---|
| 自动提取（一键提取） | 替换模式 | 清空表单，重新填充全部内容 |
| 右键提取图片 | 追加模式 | 添加图片到媒体列表，光标处插入 [图片] 占位符 |
| 右键提取元素 | 插入模式 | 在光标位置插入文本，无光标则追加末尾 |
| 选择模式提取 | 插入模式 | 在光标位置插入文本，无光标则追加末尾 |

PublishForm 提供的 API：
- `fillForm(data)` — 全量填充（替换模式）
- `insertTextAtCursor(text)` — 光标位置插入文本
- `addImages(paths)` — 追加图片到媒体列表
- `insertImagePlaceholder(n)` — 光标处插入 [图片] 占位符

### 16.8 已知限制与后续规划

**当前限制：**
- WebContentsView 是原生控件，层级高于 HTML 元素，弹窗和下拉菜单需要临时隐藏浏览器视图
- 图片下载在渲染进程中进行，大图或大量图片可能影响性能
- 手动提取的元素选择器基于 DOM 路径，页面结构变化后可能失效
- Readability.js 从 CDN 加载，需要网络连接

**后续规划：**
- 提取结果缓存（URL → 提取结果，页面内容不变时复用）
- 页面加载完成后自动预提取
- 用户反馈收集（提取准确/不准确），用于优化算法
- 更多网站适配规则（P1/P2 优先级）
- 提取质量统计面板
- 支持提取结果编辑（在填充前可修改）
