> 版本: v1.3（基于 wechat_channels 独立微服务隔离引擎）
> 适用范围: Windows 10/11，FlowX 桌面客户端 v1.1+
> 归档时间: 2026-07
> 对应源码: `src/main/services/platforms/wechat_channels.ts`

## 一、 目标页面结构分析

### 1.1 发布页 URL

微信视频号创作者助手拥有两个独立的内容发布入口，分别对应不同的媒体处理引擎：
*   **视频发布 URL**: `https://channels.weixin.qq.com/platform/post/create`
*   **图文发布 URL**: `https://channels.weixin.qq.com/platform/post/finderNewLifeCreate`

### 1.2 页面核心 DOM 结构与跨域 iframe 嵌套情报

微信视频号助手前端采用了**微前端（Micro Frontends）**架构，页面的交互元素并非呈现在主文档中，而是多层嵌套在沙箱 iframe 内。这给自动化查找和注入带来了极高的壁垒：
1.  **输入编辑器嵌套**：标题输入框 `input.weui-desktop-form__input` 和描述容器 `div.input-editor[contenteditable]` 嵌套在子 iframe 中，主页面直接执行 `document.querySelector` 无法获取到任何节点。
2.  **缩略图背景样式绑定**：图文发布上传成功后，已上传的缩略图并不挂载于普通的 `<img>` 标签下，而是作为一个 `div` 元素的背景样式存在（例如 `class="thumb-item" style="background-image: url(blob:...)"`），必须穿透提取 CSS 背景属性中的 `blob:` / `wxfile:` 以侦测上传状态。
3.  **导航误触跳转隐患**：在视频上传就绪前，DOM 中一旦检测到已存在的 `input[type="file"]`，应立即拦截前置的“模拟上传按钮点击（步骤 A）”，否则容易误触左侧导航栏中的“视频管理”选项导致整个页面跳转回列表页，发布流断裂。

### 1.3 字段字数与特殊符号限制

根据微信官方的硬性规定，对于发布内容有极其严苛的字符限制：
*   **视频模式**：短标题上限为 **16 字**；内容描述上限为 **1000 字**。
*   **图文模式**：图文标题上限为 **22 字**；内容描述上限为 **1000 字**。
*   **标题特殊符号洗涤规定**：微信视频号标题不支持大部分特殊字符（如 `|`，`-`，`*` 等），**仅支持书名号（《》）、双单引号（“”‘’"\'）、冒号（:：）、加号（+）、问号（?？）、百分号（%）和摄氏度（℃）**。中英文逗号（`,，`）在输入前必须替换为空格。

## 二、 完整发布流程

### 2.1 微信专属异步窗口构建与 UA 拦截阻塞隔离

为了彻底封锁 Electron 指纹以防被微信扫码风控拦截，同时根治时序竞争引发的“第一次登录失败”问题，视频号摒弃了通用的同步 `makePublishWindow` 逻辑，在创建 `BrowserWindow` 实例前先执行阻塞式异步指纹净化：

```
PublishEngine.submit(req)
  └─> wechat_channels.ts -> runWechatPublish()
        └─> makeWechatPublishWindow() 
              ├─> await BrowserEnvService.applyEnvironment(sess, envId) ← 100% 阻塞应用纯净 UA 与代理
              └─> new BrowserWindow()
```

### 2.2 流程运行大图

```
1. 初始化专属窗口 (makeWechatPublishWindow)
   └─> 先阻塞等待 UA 洗净及代理配置，再实例化 BrowserWindow，防范加载时序差导致 Electron 指纹泄漏重定向。
   └─> 挂载 attachNavigationTracker(win) 跟踪 did-frame-navigate 事件。

2. 加载目标发布 URL (loadURL)
   └─> 缓冲等待 350ms，以确保刚注入的 Cookie 在 Chromium 底层网络栈中完全就绪。
   └─> 执行 win.loadURL(publishUrl)。

3. 登录态验证与扫码唤醒
   └─> 检查页面是否包含 "finder-nickname" 节点或相关登录标识。
   └─> 若未登录，显示窗口提示用户完成登录操作，扫码超时时限 120 秒。
   └─> 登录成功后，若 URL 发生偏离，通过 await win.loadURL(publishUrl) 进行重载过渡并重新等待渲染稳定。

4. CDP 素材注入上传 (uploadViaCDP)
   └─> 检查页面是否存在现成的 file input 节点，有则直接跳过步骤 A 模拟点击（防止误触导航项引起跳转）。
   └─> 深度 DFS 递归遍历 DOM 树（包含 iframe 内层与 Shadow DOM）匹配 type=file 的 input。
   └─> 通过 DOM.setFileInputFiles(nodeId, files) 物理注入文件，派发 input 与 change 事件。

5. 视频后台转码缓冲静置 (防抢跑)
   └─> 素材上传完毕后，强制执行 await sleep(4500) 进行静置。
   └─> 确保微信后台服务器完成视频流切片、转码和首帧封面解析，防范过早点击导致提交被后台静默拦截。

6. 内容清洗过滤与注入 (cleanWechatTitle + 多事件失焦触发)
   └─> 调用 cleanWechatTitle() 清除不支持的特殊字符，将逗号替换为空格，截断至相应字数（视频16，图文22）。
   └─> 深度递归穿透子 iframe 定位到输入组件，通过 execCommand('insertText') 强行物理注入，完美同步 React/Vue 受控状态。
   └─> 🔑 注入完毕后触发多维度失焦事件（blur / focusout / mouseout / mouseleave），确保微信表单校验组件正确更新"已填写"状态。
   └─> 模拟鼠标移至 body 并点击，实现彻底失焦，防止因焦点仍在输入框内导致发表时校验失败。
   └─> 注入完毕后，执行 await sleep(2000) 静置，等待表单受控组件重绘渲染及内部校验刷新完成。

7. CDP 物理合成点击发表
   └─> 穿透多级 iframe 与 Shadow DOM 执行深度 DFS 定位“发表”按钮。
   └─> 通过 getAbsoluteRect 累加计算该按钮相对于外层窗口的绝对像素坐标 (x, y)。
   └─> 按钮调用 scrollIntoView 平滑滑动到视口中心。
   └─> 在主进程利用 CDP 发送 Input.dispatchMouseEvent 指令，合成 isTrusted=true 的物理鼠标点击，击穿前端防脚本风控。

8. 状态归档检测
   └─> 循环检测当前 URL 是否跳转到 /manage /list 列表页，或是否弹出“发表成功”Toast。
   └─> 成功归档后自动销毁窗口，并在 PublishEngine 中将子任务状态标记为 success。
```

## 三、 核心难点与深度解决方案

### 3.1 难点一：重试或首次发布时由于时序竞争提示未登录

**问题表现**：一键发布或在历史页重试时，第一次经常会被踢到 `login.html` 扫码页，但是不做任何修改第二次点击重试便能免扫直接进入。
**原因剖析**：Electron 窗口创建是同步的，而 `applyEnvironment` 代理设置与 UA 特征清洗是异步的。如果 `loadURL` 发起请求的时机早于异步 UA 刷入 Chromium 网络栈的耗时（时序竞争），网络首包就会携带 Electron 指纹引起风控阻断；重试成功是因为二次点击时 session 已持有前次注入完毕的 UA 缓存。
**解决方案**：
*   在 `wechat_channels.ts` 中单独实现 `makeWechatPublishWindow` 专属生成器。
*   在 `new BrowserWindow` 调用之前，使用 `await` **同步阻塞式执行 UA 净化和代理应用**。
*   将 Electron 专用的 Native 资源获取方法 `getAppIcon` 改为顶层静态导入（`import { getAppIcon } from '../../windows/MainWindow'`），对齐 Vite 静态依赖分析，根除动态 `require('./shared')` 引起的打包后模块丢失报错。

### 3.2 难点二：微信标题特殊字符限制校验拦截

**问题表现**：发布的视频或图文标题中如果含有类似 `|`，`-` 等常见分隔符，或者字数过长，发布流程进行到最终步骤时，会被官方前端组件校验拦截，导致发表按钮无法点击或表单提交无响应。
**解决方案**：
*   **正则洗涤净化**：在发布流程中接入微信视频号专属的标题净化器 `cleanWechatTitle`：
    ```typescript
    function cleanWechatTitle(title: string, maxLength: number): string {
      if (!title) return '';
      // 1. 将中英文逗号替换为单个空格
      let t = title.replace(/,|，/g, ' ');
      // 2. 正则洗净只保留中英文、数字、书名号《》、双单引号“”‘’"\'、冒号:：、加号+、问号?？、百分号%、摄氏度℃及空格
      const allowedRegex = /[^\u4e00-\u9fa5a-zA-Z0-9《》“”‘’"':：\+?？%℃\s]/g;
      t = t.replace(allowedRegex, '');
      // 3. 压缩多余空格并执行长度截断
      t = t.replace(/\s+/g, ' ').trim();
      return t.length > maxLength ? t.slice(0, maxLength) : t;
    }
    ```
*   **前端木桶效应动态自适应**：在渲染进程一键发布页面 [Publish.vue](file:///e:/WORK/flowx/flowx-desktop/src/renderer/pages/Publish.vue) 中修改了 `titleMaxLength` 计算链。当所勾选账号包含微信视频号且发布类型为 `video` 时，前端输入框最大长度会自动缩限为 **16 字**（图文为 **22 字**），在前端输入源头阻断超长风险。

### 3.3 难点三：发表按钮嵌套在 iframe / Shadow DOM 隔离树内且脚本点击不响应

**问题表现**：虽然成功定位了发表按钮并将页面滑动至按钮可见处，但是 JS 模拟的 `btnNode.click()` 和派发的 `MouseEvent('click')` 均无法使页面触发提交。
**原因剖析**：
*   **隔离树屏蔽**：发表按钮挂载在微前端架构的 Shadow DOM 和子 `<iframe>` 嵌套体内，普通 query 定位手段直接失效。
*   **Trusted 风控阻断**：微信前端安全脚本监听并校验了点击事件的 `isTrusted` 属性。脚本模拟产生的事件其该属性一律为 `false`，从而被安全模块静默丢弃。
**解决方案**：
*   **双穿透 DFS 遍历检索**：设计了能够穿透 Shadow DOM（`node.shadowRoot`）以及多层子 `<iframe>` 嵌套的深度优先遍历（DFS）算法，确保 100% 精确捕获按钮元素。
*   **跨域视口坐标偏移叠加算法**：在查找脚本中集成 `getAbsoluteRect` 偏移计算器，通过向上循环遍历祖先级 `winRef.frameElement` 的 `getBoundingClientRect` 坐标，计算出按钮在最外层主视口中的绝对物理坐标 (x, y)，像素级精准无漂移。
*   **CDP 真实物理鼠标事件模拟**：在获取坐标后，主线程通过 Chrome 调试协议的 `win.webContents.debugger.sendCommand('Input.dispatchMouseEvent')` 命令，按时序向该物理坐标发送 `mouseMoved`、`mousePressed` (按下) 及 `mouseReleased` (释放) 合成物理事件，生成完全信任的 `isTrusted === true` 凭证，完美击穿风控防线。

### 3.4 难点四：发表按钮由于过早点击导致提交无响应

**问题表现**：视频文件在界面上虽然显示上传到了 100%，但直接触发点击发表时页面没有反应。
**原因剖析**：当视频文件刚上传完毕时，微信后台仍在对视频进行分片转码、首帧封面图生成以及后台数据归档同步。如果此时填充内容并立刻进行点击，微信表单校验系统在检测到视频状态"未就绪/转码中"时会直接拒绝提交。
**解决方案**：
*   **物理延时防御**：在素材上传完成 `waitForWechatUpload` 归档后，强制令发布流进入 `await sleep(4500)` 静置状态，给视频转码和封面解析以充分的缓冲时间。
*   **表单校验重绘延迟**：在内容填充完毕后，加入 `await sleep(2000)` 静置，让 React/Vue 绑定的 state 状态在 DOM 重绘渲染并更新发表按钮激活态之后，再发起 CDP 物理点击，彻底打通发布逻辑。

### 3.5 难点五：标题填写后点击发表仍提示"未填写标题"

**问题表现**：自动化脚本已成功将标题文字填入输入框（值已写入 DOM），但点击发表按钮时，微信表单校验仍然弹出"请填写标题"或"未填写标题"的错误提示，导致发布流程被中断。该问题在视频发布和图文发布模式下均会出现。

**原因剖析**：
*   **失焦事件校验机制**：微信视频号的表单组件采用的是**失焦触发校验**的设计模式（onBlur / onFocusOut），而非输入时实时校验。只有当用户真正将焦点移出输入框时，组件才会执行内部校验逻辑并更新"已填写/未填写"状态。
*   **鼠标移出事件依赖**：部分版本的微信前端组件同时监听了 `mouseout` / `mouseleave` 鼠标移出事件来触发表单字段的脏值标记（dirty state）。仅调用 `blur()` 或派发 `change` / `input` 事件不足以完整触发校验流水线。
*   **iframe 隔离导致事件不冒泡**：标题输入框嵌套在子 iframe 的微前端沙箱中，主页面派发的事件无法穿透 iframe 边界到达输入框所在的文档上下文，导致状态同步遗漏。

**解决方案**：
*   **多维度失焦事件齐发**：在标题和描述输入完成后，一次性派发以下所有事件，确保覆盖所有可能的校验监听点：

    | 事件类型 | 事件名 | 作用 |
    |---------|--------|------|
    | 焦点事件 | `blur` | 标准失焦事件（bubbles: true） |
    | 焦点事件 | `focusout` | 焦点移出事件（支持冒泡，部分框架优先监听） |
    | 鼠标事件 | `mouseout` | 鼠标移出元素（触发表单脏值标记） |
    | 鼠标事件 | `mouseleave` | 鼠标离开元素（部分 UI 库专用） |
    | 原生方法 | `element.blur()` | 调用 DOM 原生 blur 方法，移除 activeElement |

*   **模拟彻底失焦**：在输入框 blur 之后，额外对 `document.body` 派发 `mouseover` 事件并执行 `body.click()`，模拟用户点击页面空白处的行为，确保焦点完全从输入框移走，触发最外层的表单校验。

*   **iframe 穿透执行**：所有事件派发均在找到输入框的同一 document 上下文中执行（通过 `findElements` 递归进入 iframe 后直接操作该 iframe 内的元素），避免了跨 iframe 事件不穿透的问题。

*   **静置等待校验完成**：填充和失焦操作完成后，保持 `await sleep(2000)` 的静置时间，给微信表单组件充足的时间执行内部校验状态刷新和 UI 重绘。

## 四、 核心实现代码展示

### 4.1 双穿透及视口坐标叠加提取脚本 (buildWechatClickPublishScript)

```typescript
function buildWechatClickPublishScript(): string {
  return `
    (function() {
      var r = { clicked: false, err: '', buttonText: '', x: 0, y: 0 };
      try {
        // 🔑 叠加计算嵌套 iframe 隔离树下元素的绝对物理像素视口坐标
        function getAbsoluteRect(node) {
          var rect = node.getBoundingClientRect();
          var left = rect.left;
          var top = rect.top;
          var winRef = node.ownerDocument.defaultView;
          while (winRef && winRef !== window) {
            try {
              var frameElement = winRef.frameElement;
              if (frameElement) {
                var fRect = frameElement.getBoundingClientRect();
                left += fRect.left;
                top += fRect.top;
              }
            } catch(e) {
              break;
            }
            winRef = winRef.parent;
          }
          return { left: left, top: top, width: rect.width, height: rect.height };
        }

        // 🔑 深度 DFS 递归穿透普通 DOM、Shadow DOM 与 Iframe
        function findPublishButton(doc) {
          if (!doc) return null;
          
          function search(node) {
            if (!node) return null;
            
            // 1. 若是按钮节点，按文字内容进行规则匹配
            if (node.nodeType === 1) {
              var tagName = node.tagName.toLowerCase();
              var role = node.getAttribute && node.getAttribute('role') || '';
              var cls = node.getAttribute && node.getAttribute('class') || '';
              
              if (tagName === 'button' || role === 'button' || cls.indexOf('btn') !== -1 || cls.indexOf('button') !== -1) {
                if (!(node.disabled || node.getAttribute('disabled') === 'true' || cls.indexOf('disabled') !== -1)) {
                  var txt = (node.innerText || node.textContent || '').replace(/\\s+/g, '').trim();
                  if (txt === '发表' || txt === '发布' || txt === '发表视频' || txt === '发表图文') {
                    return node;
                  }
                }
              }
            }
            
            // 2. 递归子节点
            if (node.childNodes && node.childNodes.length > 0) {
              for (var i = 0; i < node.childNodes.length; i++) {
                var res = search(node.childNodes[i]);
                if (res) return res;
              }
            }
            
            // 3. 递归 Shadow Root
            if (node.shadowRoot) {
              var res = search(node.shadowRoot);
              if (res) return res;
            }
            
            // 4. 递归 Iframe
            if (node.tagName && node.tagName.toLowerCase() === 'iframe') {
              try {
                var idoc = node.contentDocument || (node.contentWindow && node.contentWindow.document);
                if (idoc) {
                  var res = search(idoc);
                  if (res) return res;
                }
              } catch(e) {}
            }
            
            return null;
          }
          
          return search(doc.body || doc);
        }

        var btnNode = findPublishButton(document);
        if (btnNode) {
          try {
            btnNode.scrollIntoView({ block: 'center', inline: 'nearest' });
          } catch(e) {}
          
          var absRect = getAbsoluteRect(btnNode);
          r.x = Math.round(absRect.left + absRect.width / 2);
          r.y = Math.round(absRect.top + absRect.height / 2);

          // 物理点击兜底
          btnNode.click();
          btnNode.dispatchEvent(new MouseEvent('click', { bubbles: true, cancelable: true }));
          r.clicked = true;
          r.buttonText = btnNode.innerText || btnNode.textContent || '';
        } else {
          r.err = '未能在页面或 iframe 中找到发表按钮';
        }
      } catch(e) {
        r.err = e.message;
      }
      return r;
    })()
  `;
}
```

### 4.2 主线程 CDP 物理鼠标事件合成触发 (Input.dispatchMouseEvent)

```typescript
// 在 runWechatPublish 中检测到 findPublishButton 返回绝对坐标后执行
if (clickRes.x > 0 && clickRes.y > 0) {
  log('info', 'publish', `准备通过 CDP 发送物理鼠标点击至坐标: (${clickRes.x}, ${clickRes.y})`);
  try {
    // 1. 移动鼠标到目标中心位置
    await win.webContents.debugger.sendCommand('Input.dispatchMouseEvent', {
      type: 'mouseMoved', x: clickRes.x, y: clickRes.y, button: 'left', clickCount: 1, buttons: 1, modifiers: 0,
    });
    await sleep(80);
    
    // 2. 鼠标左键按下 (mousePressed)
    await win.webContents.debugger.sendCommand('Input.dispatchMouseEvent', {
      type: 'mousePressed', x: clickRes.x, y: clickRes.y, button: 'left', clickCount: 1, buttons: 1, modifiers: 0,
    });
    await sleep(120);
    
    // 3. 鼠标左键抬起释放 (mouseReleased)
    await win.webContents.debugger.sendCommand('Input.dispatchMouseEvent', {
      type: 'mouseReleased', x: clickRes.x, y: clickRes.y, button: 'left', clickCount: 1, buttons: 0, modifiers: 0,
    });
    log('info', 'publish', '✅ CDP 真实鼠标物理点击发送成功');
  } catch(cdpErr: any) {
    log('warn', 'publish', `CDP 鼠标物理点击发送失败: ${cdpErr.message}`);
  }
}
```

## 五、 稳定性保障机制

1.  **安全脚本执行工具 (safeExecuteJavaScript)**：针对微信微前端中部分微服务 SSL 连接报错导致 `webContents.executeJavaScript` 原生 Promise 挂死的问题，封装了包含 4000ms 超时限制的 `Promise.race` 包装层，为所有轮询与检测上了“防卡死安全锁”。
2.  **前置 input file 拦截机制**：在 CDP 上传前探测主文档和 iframe，存在现成 file input 则直接跳过步骤 A 的按钮模拟点击，彻底解决了误碰侧栏导致页面跳转的 Bug。
3.  **转码防抢跑延时 (sleep)**：对上传完毕后的状态等待和填充后的重绘校验分别设置了 4.5 秒与 2.0 秒的硬性物理等待时间，完美确保微信后台与 React 表单处于最终态，为每一次自动化发布保航。
