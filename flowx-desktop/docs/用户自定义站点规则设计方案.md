## 1. 功能概述

用户自定义站点规则允许进阶用户为常用网站创建专属的内容提取规则，当访问匹配的网站时自动使用用户规则，获得比通用算法更精准的提取效果。

> 💡 **核心价值**
> - **对用户**：常用网站提取效果可调优，提取结果更可控
> - **对产品**：降低对内置规则的依赖，扩大自动提取覆盖范围
> - **对生态**：规则可导入导出，用户间可分享，形成社区效应

### 1.1 功能模块组成

| 模块 | 说明 |
|------|------|
| 📋 **规则管理** | 规则列表展示、新增/编辑/删除、启用/禁用开关、搜索过滤。 |
| ✏️ **规则编辑器** | 表单化编辑各选择器字段，配合实时预览测试，所见即所得。 |
| ⚡ **一键保存** | 手动提取后一键保存为站点规则，自动填充选择器，零代码门槛。 |
| 📤 **导入导出** | JSON 格式导入导出单条或全部规则，方便备份和分享。 |

### 1.2 与现有功能的关系

**图 1：用户自定义规则在提取体系中的位置**

```
用户自定义规则 → SiteRuleManager → ContentExtractor → 三级提取策略
                                                          ↓
                                         ┌───────────────────────────┐
                                         │ 1. 自定义规则（最高优先级）│
                                         │ 2. 内置站点规则            │

                                         └───────────────────────────┘

手动提取/选择提取 →（一键保存）→ 用户自定义规则
规则编辑器 →（可视化编辑）→ 用户自定义规则
导入/导出 →（JSON）→ 用户自定义规则
```

## 2. 数据模型设计

### 2.1 自定义规则数据结构

自定义规则采用**可序列化的 JSON 结构**，便于存储和导入导出。与现有内部 `SiteRule` 接口区分开，内部使用时再转换。

```typescript
/**
 * 用户自定义站点规则（可序列化，存储用）
 */
export interface CustomSiteRule {
  // ===== 基本信息 =====
  /** 规则唯一 ID（UUID） */
  id: string;
  /** 规则名称，用户可自定义 */
  name: string;
  /** 是否启用 */
  enabled: boolean;
  /** 创建时间 */
  createdAt: number;
  /** 最后更新时间 */
  updatedAt: number;

  // ===== 匹配规则 =====
  /** 匹配方式：domain（域名包含） / regex（正则表达式） */
  matchType: 'domain' | 'regex';
  /** 匹配值：domain 模式下是域名字符串，regex 模式下是正则表达式字符串 */
  matchValue: string;
  /** 路径匹配（可选，进一步限制路径范围） */
  pathPattern?: string;

  /**
   * 适用的发布类型（可多选）
   * - image-text: 图文发布
   * - video: 视频发布
   * - article: 文章发布
   * 为空数组表示适用于所有类型
   */
  contentTypes: Array<'image-text' | 'video' | 'article'>;

  // ===== 提取规则 =====
  /** 标题选择器（CSS 选择器，可为空） */
  titleSelector?: string;
  /** 正文选择器（CSS 选择器，必填） */
  contentSelector: string;
  /** 作者/来源选择器（可选） */
  bylineSelector?: string;
  /** 发布时间选择器（可选） */
  dateSelector?: string;
  /** 站点名称（可选，不填则自动从页面获取） */
  siteName?: string;
  /** 图片选择器（可选，不填则从正文内自动查找） */
  imageSelector?: string;
  /** 话题/标签选择器（可选，提取文章的话题标签列表） */
  tagsSelector?: string;

  /** 需要移除的噪音元素选择器列表 */
  removeSelectors: string[];

  // ===== 元数据 =====
  /** 规则备注/说明 */
  remark?: string;
  /** 规则来源：manual（手动创建）/ quick-save（一键保存）/ import（导入） */
  source: 'manual' | 'quick-save' | 'import';
  /** 使用次数统计 */
  useCount: number;
  /** 最后使用时间 */
  lastUsedAt?: number;
}
```

### 2.2 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `matchType` | 'domain' \| 'regex' | 是 | **domain 模式**：简单的域名包含匹配，如 `example.com`，适用于大多数场景，用户无需了解正则。<br/>**regex 模式**：完整的正则表达式匹配，如 `/blog\.example\.com\/post\//i`，适用于复杂场景。 |
| `matchValue` | string | 是 | 匹配值。domain 模式下是域名字符串；regex 模式下是正则表达式字符串（不含两端的 / 和 flag）。 |
| `pathPattern` | string | 否 | 可选的路径匹配正则，进一步缩小规则适用范围。仅在域名匹配后校验。 |
| `contentTypes` | string[] | 否 | **适用的发布类型**，可多选：`image-text`（图文）、`video`（视频）、`article`（文章）。空数组表示适用于所有类型。右键菜单中按当前发布类型优先展示匹配的规则。 |
| `contentSelector` | string | 是 | 正文容器的 CSS 选择器。规则的核心字段。 |
| `removeSelectors` | string[] | 否 | 需要从正文中移除的噪音元素选择器列表，如广告、推荐、评论区等。 |
| `enabled` | boolean | 是 | 是否启用该规则。禁用的规则不会参与匹配。默认 true。 |

### 2.3 存储方案

存储在 SecureStore 中，新增 `customSiteRules` 字段：

```typescript
// SecureStore Schema 新增字段
interface StoreSchema {
  // ... 现有字段 ...

  /** 用户自定义站点规则 */
  customSiteRules: CustomSiteRule[];
}

// 默认值：空数组
defaults: {
  // ...
  customSiteRules: [],
}
```

> 📐 **容量限制**
>
> 最多保存 **100 条**自定义规则。超出时提示用户删除旧规则。单条规则约 0.5-1KB，100 条约 50-100KB，对存储无压力。

## 3. 规则管理器

### 3.1 设计思路

将现有的硬编码 `SITE_RULES` + `findSiteRule()` 模式，重构为**统一的 SiteRuleManager**，统一管理内置规则和自定义规则，提供一致的 CRUD 接口。

### 3.2 管理器类结构

```typescript
/**
 * 站点规则管理器
 * 统一管理内置规则和用户自定义规则
 */
class SiteRuleManager {
  // 内置规则（从硬编码迁移，或从 JSON 文件加载）
  private builtInRules: SiteRule[] = [];

  // 用户自定义规则
  private customRules: CustomSiteRule[] = [];

  // ===== 初始化 =====

  /**
   * 初始化：加载内置规则 + 用户自定义规则
   */
  async initialize(): Promise<void>;

  /**
   * 从 SecureStore 加载自定义规则
   */
  private loadCustomRules(): void;

  /**
   * 保存自定义规则到 SecureStore
   */
  private saveCustomRules(): void;

  // ===== 匹配查询 =====

  /**
   * 根据 URL 查找匹配的规则
   * 优先级：自定义规则（启用的）→ 内置规则
   * 返回可直接用于提取的 SiteRule 对象
   */
  findMatchingRule(url: string): SiteRule | null;

  /**
   * 检查 URL 是否匹配某条自定义规则
   */
  private matchCustomRule(rule: CustomSiteRule, url: string): boolean;

  // ===== 自定义规则 CRUD =====

  /**
   * 获取所有自定义规则
   */
  getCustomRules(): CustomSiteRule[];

  /**
   * 新增自定义规则
   */
  addCustomRule(rule: Omit<CustomSiteRule, 'id' | 'createdAt' | 'updatedAt' | 'useCount'>): CustomSiteRule;

  /**
   * 更新自定义规则
   */
  updateCustomRule(id: string, updates: Partial<CustomSiteRule>): CustomSiteRule | null;

  /**
   * 删除自定义规则
   */
  deleteCustomRule(id: string): boolean;

  /**
   * 切换规则启用状态
   */
  toggleRule(id: string, enabled?: boolean): boolean;

  /**
   * 复制一条规则
   */
  duplicateRule(id: string): CustomSiteRule | null;

  // ===== 批量操作 =====

  /**
   * 批量导入规则（去重：同名或同匹配模式的合并）
   * 返回实际导入的数量
   */
  importRules(rules: CustomSiteRule[], mode: 'merge' | 'replace'): number;

  /**
   * 导出所有自定义规则
   */
  exportRules(ids?: string[]): CustomSiteRule[];

  // ===== 统计 =====

  /**
   * 记录规则使用（每次成功匹配后调用）
   */
  recordUsage(id: string): void;
}
```

### 3.3 匹配优先级

| 优先级 | 规则类型 | 说明 |
|--------|----------|------|
| **1（最高）** | 用户自定义-启用 | 用户创建且已启用的规则，优先级最高。有多条匹配时按更新时间排序，最新的优先。 |
| **2** | 内置规则 | 官方预置的站点规则，在 SITE_RULES 中定义。 |
| **3** | 用户自定义-禁用 | 用户创建但已禁用的规则，不参与自动匹配，仅在规则列表中可见。 |

### 3.4 matchType 的匹配实现

```typescript
private matchCustomRule(rule: CustomSiteRule, url: string): boolean {
  if (!rule.enabled) return false;

  try {
    // 1. 域名/URL 主体匹配
    let matched = false;

    if (rule.matchType === 'domain') {
      // domain 模式：检查 URL 的 hostname 是否包含指定域名
      const hostname = new URL(url).hostname;
      matched = hostname === rule.matchValue || hostname.endsWith('.' + rule.matchValue);
    } else {
      // regex 模式：用正则表达式匹配完整 URL
      // 为安全起见，对用户输入的正则做 try-catch
      const regex = new RegExp(rule.matchValue, 'i');
      matched = regex.test(url);
    }

    if (!matched) return false;

    // 2. 路径匹配（可选）
    if (rule.pathPattern) {
      const pathname = new URL(url).pathname;
      const pathRegex = new RegExp(rule.pathPattern, 'i');
      if (!pathRegex.test(pathname)) return false;
    }

    return true;
  } catch (e) {
    // 正则语法错误等异常，视为不匹配
    logger.warn(`[SiteRuleManager] 规则匹配异常: ${rule.name}`, e);
    return false;
  }
}
```

> ⚠️ **安全注意**
>
> 用户输入的正则表达式用 `try-catch` 包裹，防止语法错误导致崩溃。正则只在主进程中执行（通过 executeJavaScript 注入到页面），不涉及用户数据安全。

### 3.5 自定义规则 → SiteRule 转换

```typescript
/**
 * 将 CustomSiteRule 转换为提取用的 SiteRule
 */
private toSiteRule(customRule: CustomSiteRule): SiteRule {
  let match: RegExp;

  if (customRule.matchType === 'domain') {
    // 转义域名中的特殊字符，构建包含匹配的正则
    const escaped = customRule.matchValue.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
    match = new RegExp(`(^|\\.)${escaped}`, 'i');
  } else {
    match = new RegExp(customRule.matchValue, 'i');
  }

  return {
    name: `custom:${customRule.id}`,
    match,
    titleSelector: customRule.titleSelector,
    contentSelector: customRule.contentSelector,
    bylineSelector: customRule.bylineSelector,
    dateSelector: customRule.dateSelector,
    siteName: customRule.siteName,
    imageSelector: customRule.imageSelector,
    removeSelectors: customRule.removeSelectors || [],
  };
}
```

## 4. 与提取流程集成

### 4.1 集成点

修改 `extractContentFromView` 函数，将原来的 `findSiteRule()` 替换为 `SiteRuleManager.findMatchingRule()`，同时在提取成功后记录使用统计。

```typescript
// 修改前
export async function extractContentFromView(viewId: string): Promise<ExtractedContent | null> {
  // ...
  const siteRule = findSiteRule(pageUrl);  // 只查内置规则
  // ...
}

// 修改后
export async function extractContentFromView(viewId: string): Promise<ExtractedContent | null> {
  // ...
  const siteRule = siteRuleManager.findMatchingRule(pageUrl);  // 统一查自定义+内置
  if (siteRule) {
    const result = await trySiteRuleExtract(wc, siteRule, pageUrl);
    if (result) {
      // 记录自定义规则使用统计
      if (siteRule.name.startsWith('custom:')) {
        const ruleId = siteRule.name.replace('custom:', '');
        siteRuleManager.recordUsage(ruleId);
      }
      return result;
    }
  }
  // 继续回退到 Readability...
  // ...
}
```

### 4.2 置信度调整

用户自定义规则的置信度与内置站点规则保持一致（基础 85 分），但策略标识不同：

| 策略 | extractStrategy 值 | 基础置信度 | UI 标签颜色 |
|------|-------------------|------------|-------------|
| 内置站点规则 | `'site-rule'` | 85 | 绿色 |
| **用户自定义规则** | `'custom-rule'` | **85** | **紫色** |
| Readability | `'readability'` | 50 | 蓝色 |
| 自动算法 | `'auto'` | 40 | 黄色 |
| 手动提取 | `'manual'` | 90 | 绿色 |

> 🎨 **UI 表现**
>

## 5. 浏览器规则面板

### 5.1 设计背景

原设计中规则编辑器采用**右侧抽屉**形式，但由于 Electron 中 `WebContentsView` 是原生视图层级，位于渲染进程 DOM 之上，导致抽屉弹窗会被浏览器页面遮挡（z-index 问题）。

> ⚠️ **原生视图层级问题**
>
> Electron 的 `BrowserView` / `WebContentsView` 是原生窗口级别，不受 CSS z-index 控制。任何渲染进程中的 DOM 元素都无法覆盖在原生视图之上。因此抽屉、弹窗、Tooltip 等覆盖式 UI 在浏览器区域内无法正常显示。

### 5.2 解决方案：标签页式右侧面板

采用**并排布局 + 标签页切换**的方案，将规则管理界面与发布编辑面板整合到右侧面板中，通过 Tab 切换，从根本上避免 z-index 问题。

**浏览器页面布局（分屏模式）**

```
┌─────────────────────────────────────────────────────────────────┐
│  🔍 地址栏                                [⚙️规则] [📐布局]    │
├──────────────────────────┬──────────────────────────────────────┤
│                          │  📝 发布编辑    ⚙️ 提取规则  [Tab]   │
│                          ├──────────────────────────────────────┤
│                          │                                      │
│    浏览器 WebContents    │   右侧面板内容区                      │
│    (原生视图)            │   - 发布表单 / 规则列表               │
│                          │   - 规则编辑器                       │
│                          │   - 拾取器状态提示                   │
│                          │                                      │
│                          │                                      │
└──────────────────────────┴──────────────────────────────────────┘
```

### 5.3 右侧面板结构

| 面板 | 说明 |
|------|------|
| 📝 **发布编辑 Tab** | 默认显示，包含发布表单、草稿保存、测试提交等功能。 |
| ⚙️ **提取规则 Tab** | 规则列表、新建/编辑规则、可视化拾取、测试应用等功能。 |

### 5.4 交互流程

**入口触发**
- 点击工具栏「提取规则」按钮 → 展开右侧面板并切换到规则 Tab
- 右键菜单「添加自定义规则」 → 自动展开面板 + 切换到规则 Tab + 打开编辑器
- 右键菜单「使用自定义规则提取」 → 自动切换到发布 Tab 并填入内容

**拾取器联动**
1. 用户在规则编辑器中点击「拾取」按钮
2. 拾取器启动，鼠标在浏览器页面上移动时高亮元素
3. 拾取结果实时显示在右侧规则面板中（无需弹窗）
4. 用户可继续拾取其他字段或保存规则

### 5.5 组件结构


### 5.6 布局模式

支持三种布局模式，用户可根据需要切换：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **browser-only** | 仅显示浏览器，右侧面板完全隐藏 | 纯浏览网页时 |
| **split** | 左右分屏，浏览器 + 右侧面板并排显示 | 边浏览边编辑 / 配置规则时（默认） |
| **publish-only** | 仅显示右侧面板，浏览器隐藏 | 专注编辑发布内容时 |

> ✅ **方案优势**
> - **彻底解决 z-index 问题**：所有 UI 都在渲染进程 DOM 内，无原生视图层级冲突
> - **操作效率更高**：规则配置与发布编辑同屏切换，无需打开关闭弹窗
> - **视觉一致性**：与发布表单统一在右侧面板，用户学习成本低
> - **多任务并行**：可边浏览网页边配置规则，实时验证效果

## 8. 可视化元素拾取器

### 8.1 设计目标

**零代码配置规则**。用户无需了解 CSS 选择器，只需在页面上点击对应的元素，系统自动生成选择器并填入规则表单。支持单选元素（标题、正文、作者等）和多选元素（图片列表、话题标签、移除元素等）两种模式。

> ✅ **核心能力**
> - **单选模式**：点击一个元素，生成精确的 CSS 选择器
> - **多选模式**：点击多个同类元素，自动推断通用选择器（如 `.tag-list .tag`）
> - **实时预览**：高亮当前选中的元素，显示选择器和文本预览
> - **层级调整**：上下方向键在 DOM 层级中移动，选择父/子元素

### 8.5 多选模式的通用选择器推断

当用户选择 2 个以上同类元素时，系统需要自动推断出能够匹配所有已选元素的**最精确的通用选择器**。推断算法按以下优先级尝试：

1. **相同 class + 相同父级**：如果所有已选元素有相同的 class，且父元素也有相同的 class，生成 `.parent-class .child-class` 形式的选择器（最优先，最稳定）
2. **相同 tag + 相同 class**：如果所有已选元素有相同的标签名和 class，生成 `tag.classname` 形式
3. **相同 class**：如果所有已选元素有共同的 class，生成 `.common-class` 形式
4. **相同标签 + 父级选择器**：如果没有共同 class，但有相同的标签名，找最近的共同祖先，生成 `.ancestor tagname` 形式
5. **兜底：属性选择器**：如果以上都不匹配，使用共同的属性（如 `[data-tag]`）

**推断示例**

| 场景 | 用户点击的元素 | 推断出的通用选择器 |
|------|---------------|-------------------|
| 话题标签 | `<a class="tag-item">科技</a>`<br/>`<a class="tag-item">AI</a>` | `.tag-item` |
| 文章图片 | `<div class="content"><img ...></div>` 中的两张图 | `.content img` |
| 评论区移除 | `<div id="comment-section" ...>` | `#comment-section` |
| 推荐列表 | `<div class="related-posts">`<br/>`<div class="recommend-box">` | （无通用选择器，分别添加） |

> ⚠️ **推断失败处理**
>
> 如果无法推断出通用选择器（如两个元素完全不相关），提示用户"所选元素类型不同"，允许用户继续添加更多元素，或手动分别添加为独立的移除选择器。

### 9.5 自动填充逻辑

保存对话框中，以下字段自动填充，用户可按需调整：

| 字段 | 填充值来源 | 是否可编辑 |
|------|-----------|------------|
| 规则名称 | 默认 `"{网站域名} 规则"`，用户可修改 | 是 |
| 匹配方式 | 默认 `domain` 模式 | 否（高级编辑中可改） |
| 匹配域名 | 从当前页面 URL 提取 hostname | 只读 |
| 正文选择器 | 手动提取时使用的 CSS 选择器 | 只读（高级编辑中可改） |
| 标题选择器 | 不填充（留空，自动获取页面标题） | - |
| 移除元素 | 不填充（空列表） | - |

> 💡 **设计思路**
>
> 一键保存的目标是**快速创建、零门槛**，因此只暴露最核心的字段（名称 + 域名 + 正文选择器）。高级选项（标题选择器、移除元素等）引导用户到完整的规则编辑器中调整。这样既降低了使用门槛，又保留了进阶能力。

### 9.6 重复规则检测

保存时检测是否已存在相同域名的自定义规则：

- 如果已存在：提示"该网站已有一条自定义规则，是否覆盖？"，提供"覆盖" / "创建新规则" / "取消" 三个选项
- 如果不存在：直接保存

## 10. 右键菜单快捷规则

### 10.1 设计目标

在浏览器页面的右键菜单中增加**"使用自定义规则提取"**子菜单，让用户可以快速选择某条自定义规则来提取当前页面内容，无需打开规则编辑器或设置页，提升操作效率。

> ✅ **核心价值**
> - **快速应用**：右键即可选择规则，一步到位
> - **智能排序**：匹配当前网站的规则优先展示
> - **类型匹配**：根据当前发布类型（图文/视频/文章）优先展示对应规则
> - **全量可及**：所有自定义规则都能在二级菜单中找到

### 10.2 菜单结构

### 10.2.1 使用规则 vs 添加规则

右键菜单中包含两个与自定义规则相关的入口，分别对应不同的使用场景：

| 菜单项 | 图标 | 场景 | 行为 |
|--------|------|------|------|
| **使用自定义规则提取** | 📋 | 已有规则，想快速应用 | 选择规则 → 直接提取当前页面 → 填充表单 |
| **添加自定义规则** | ➕ | 没有规则，想从页面创建 | 选择拾取类型 → 在页面上拾取元素 → 保存为新规则 |

### 10.3 菜单排序逻辑

菜单项的排序遵循以下优先级，确保用户最可能用到的规则排在最前面：

1. **第一优先级：当前网站匹配 + 发布类型匹配**
   域名匹配且 contentTypes 包含当前发布类型的规则，排在最前面
2. **第二优先级：当前网站匹配**
   域名匹配但 contentTypes 不匹配（或为空）的规则
3. **第三优先级：发布类型匹配**
   域名不匹配但 contentTypes 匹配当前发布类型的规则
4. **第四优先级：其他规则**
   既不匹配域名也不匹配类型的规则，放在"全部自定义规则"二级菜单中

**排序示例**

| 场景 | 当前页面 | 当前发布类型 | 菜单展示（从上到下） |
|------|---------|-------------|---------------------|
| **场景1** | blog.example.com | 图文 | 1. ⭐ 我的博客图文规则（域名+类型都匹配）<br/>2. 📝 我的博客通用规则（仅域名匹配）<br/>3. 📋 全部自定义规则 → ... |
| **场景2** | news.example.com | 文章 | 1. 📰 新闻文章规则（域名+类型都匹配）<br/>2. 📋 全部自定义规则 → ... |
| **场景3** | 未知网站 | 视频 | （无匹配规则，直接显示二级菜单入口）<br/>1. 📋 全部自定义规则 → 视频平台规则、... |

### 10.4 发布类型感知

右键菜单需要知道当前的发布类型，才能进行类型匹配。发布类型从渲染进程的发布表单中获取。

**获取当前发布类型的方式**

- 渲染进程维护当前发布类型状态（图文/视频/文章）
- 切换发布类型时，通过 IPC 通知主进程更新
- 主进程在构建右键菜单时使用当前发布类型进行排序
- 如果无法获取发布类型（如发布表单未打开），则忽略类型匹配，只按域名排序

### 10.5 规则图标与标签

每条规则在菜单中显示图标和标签，帮助用户快速识别：

| 元素 | 显示 | 说明 |
|------|------|------|
| **规则名称** | 主标题文字 | 用户自定义的规则名称 |
| **类型图标** | 📝 图文 / 🎬 视频 / 📰 文章 / 🌐 通用 | 根据 contentTypes 显示适用的发布类型图标 |
| **匹配标记** | ⭐ | 匹配当前网站的规则前加星标 |
| **使用次数** | （tooltip 中显示） | 悬停时显示使用次数和最后使用时间 |

### 10.6 点击菜单项后的行为

1. 用户点击某条规则
2. 主进程调用 `trySiteRuleExtract()`，使用该规则提取当前页面
3. 提取成功后，通过 `browser:extractResult` 事件将结果推送到渲染进程
4. 渲染进程填充发布表单（与一键提取相同的填充逻辑）
5. 记录该规则的使用次数和最后使用时间
6. 置信度标签显示"自定义规则"（紫色）

> 💡 **与一键提取的区别**
>
> 一键提取走三级回退策略（自定义规则 → 内置规则 → Readability → 文本密度），而右键菜单选择指定规则则**强制使用该规则**，不回退。如果规则提取失败，提示用户"该规则在此页面提取失败"，用户可以换一条规则或用一键提取。

### 10.7 菜单构建时机

右键菜单是在用户每次右键时**动态构建**的，确保菜单内容始终与当前页面和规则列表同步。

```typescript
// 构建右键菜单中的自定义规则部分（伪代码）
function buildCustomRuleMenu(viewId, pageUrl, publishType) {
  const allRules = siteRuleManager.getCustomRules().filter(r => r.enabled);

  // 分类
  const siteMatchTypeMatch = [];  // 域名+类型都匹配
  const siteMatchOnly = [];        // 仅域名匹配
  const typeMatchOnly = [];        // 仅类型匹配
  const others = [];               // 都不匹配

  for (const rule of allRules) {
    const matchesSite = matchesRule(rule, pageUrl);
    const matchesType = rule.contentTypes.length === 0
      || rule.contentTypes.includes(publishType);

    if (matchesSite && matchesType) siteMatchTypeMatch.push(rule);
    else if (matchesSite) siteMatchOnly.push(rule);
    else if (matchesType) typeMatchOnly.push(rule);
    else others.push(rule);
  }

  // 按使用次数排序（常用的在前）
  const sortByUsage = (a, b) => (b.useCount || 0) - (a.useCount || 0);
  siteMatchTypeMatch.sort(sortByUsage);
  siteMatchOnly.sort(sortByUsage);
  typeMatchOnly.sort(sortByUsage);
  others.sort(sortByUsage);

  // 构建菜单项
  const menuItems = [];

  // 第一组：域名+类型都匹配
  if (siteMatchTypeMatch.length > 0) {
    menuItems.push({ type: 'separator' });
    siteMatchTypeMatch.forEach(rule => {
      menuItems.push({
        label: `⭐ ${rule.name}`,
        click: () => applyRule(viewId, rule),
      });
    });
  }

  // 第二组：仅域名匹配
  if (siteMatchOnly.length > 0) {
    if (menuItems.length > 0) menuItems.push({ type: 'separator' });
    siteMatchOnly.forEach(rule => {
      menuItems.push({
        label: rule.name,
        click: () => applyRule(viewId, rule),
      });
    });
  }

  // 第三组：全部自定义规则（二级菜单）
  const allSorted = [...siteMatchTypeMatch, ...siteMatchOnly, ...typeMatchOnly, ...others];
  const allUnique = [...new Map(allSorted.map(r => [r.id, r])).values()];

  if (allUnique.length > 0) {
    if (menuItems.length > 0) menuItems.push({ type: 'separator' });
    menuItems.push({
      label: '📋 全部自定义规则',
      submenu: allUnique.map(rule => ({
        label: rule.name,
        click: () => applyRule(viewId, rule),
      })),
    });
  }

  return menuItems;
}
```

### 10.8 空状态处理

- **没有自定义规则时**：不显示"使用自定义规则提取"菜单项
- **所有规则都禁用时**：同上，不显示
- **只有 1 条匹配规则时**：直接显示在一级，不展开二级
- **匹配规则超过 5 条时**：前 5 条放一级，剩余的放二级"更多匹配规则"

### 10.9 添加自定义规则

在右键菜单中增加**"➕ 添加自定义规则"**子菜单，让用户直接从当前页面开始创建规则，无需先打开设置页或规则编辑器。子菜单提供各种元素拾取入口，点击后直接进入拾取模式。

**子菜单结构**

| 菜单项 | 图标 | 拾取模式 | 说明 |
|--------|------|----------|------|
| 拾取标题元素 | 📌 | 单选 | 点击页面上的标题元素，自动填入标题选择器 |
| 拾取正文元素 | 📝 | 单选 | 点击页面上的正文容器，自动填入正文选择器 |
| 拾取图片元素 ▶ | 🖼️ | 多选 | 二级菜单：开始多选拾取 / 拾取单张图片 |
| 拾取话题标签 ▶ | 🏷️ | 多选 | 二级菜单：开始多选拾取 / 拾取单个标签 |
| 拾取作者元素 | 👤 | 单选 | 点击页面上的作者名元素 |
| 拾取日期元素 | 📅 | 单选 | 点击页面上的发布时间元素 |
| 拾取移除元素 ▶ | 🗑️ | 逐个添加 | 二级菜单：开始拾取移除元素 / 清除已移除 |
| **完整规则编辑器** | ✏️ | - | 打开规则编辑器抽屉，从当前网站创建一条新规则 |

**快速拾取流程（从右键菜单直接拾取）**

1. 用户在页面上右键 → 选择"添加自定义规则" → 选择"拾取正文元素"
2. 拾取器启动，进入正文拾取模式（单选）
3. 用户在页面上点击正文容器元素
4. 拾取完成，弹出**快速保存弹窗**：
   - 规则名称（自动填充："xxx.com 规则"，可修改）
   - 匹配域名（自动从当前 URL 提取）
   - 已拾取的字段（正文选择器：已填充）
   - 按钮：**继续拾取其他字段** / **保存规则**
5. 如果点击"继续拾取其他字段"：关闭弹窗，保持拾取器，可继续从右键菜单选择其他拾取类型
6. 如果点击"保存规则"：直接保存为新规则，提示"规则已保存"，并自动用该规则提取当前页面

> 💡 **增量拾取设计**
>
> 快速拾取支持**增量模式**：一次拾取会话中可以依次拾取多个字段。拾取器在同一次会话中持续激活，每次拾取一个字段后追加到当前规则草稿中。用户可以随时从右键菜单切换拾取类型，直到点击"保存规则"完成创建。

**草稿规则的临时存储**

从右键菜单拾取元素时，规则还未保存，处于"草稿"状态。草稿数据存储在渲染进程的内存中：

```typescript
// 草稿规则数据结构
interface RuleDraft {
  name: string;                    // 规则名称（自动生成）
  matchType: 'domain';             // 固定 domain 模式
  matchValue: string;              // 从当前 URL 提取
  contentTypes: string[];          // 默认为空（所有类型）
  titleSelector?: string;          // 拾取标题后填充
  contentSelector?: string;        // 拾取正文后填充
  bylineSelector?: string;         // 拾取作者后填充
  dateSelector?: string;           // 拾取日期后填充
  imageSelector?: string;          // 拾取图片后填充
  tagsSelector?: string;           // 拾取话题标签后填充
  removeSelectors: string[];       // 拾取移除元素后填充
  pickerSessionActive: boolean;    // 拾取会话是否进行中
  lastPickerType?: string;         // 最后一次拾取的字段类型
}
```

**草稿状态的 UI 提示**

拾取会话进行中，页面底部显示浮动提示栏，展示当前草稿状态：

**底部浮动提示栏**

```
┌──────────────────────────────────────────────────────────┐
│  ✏️ 正在创建规则：example.com 规则     [已拾取 2/7 字段] │
│  📌 标题：已拾取    📝 正文：已拾取    🖼️ 图片：未拾取  │
│                                                          │
│  [ 继续拾取 ▼ ]   [ 测试规则 ]   [ 保存规则 ]   [ 取消 ] │
└──────────────────────────────────────────────────────────┘
```

- **继续拾取 ▼**：下拉菜单，选择下一个要拾取的字段类型
- **测试规则**：用当前已拾取的选择器测试提取效果
- **保存规则**：保存草稿为正式规则，结束拾取会话
- **取消**：丢弃草稿，退出拾取模式

**完整规则编辑器入口**

如果用户觉得快速拾取不够用，可以从子菜单底部选择**"✏️ 完整规则编辑器"**，直接打开规则编辑器抽屉，创建一条新规则。新规则会自动填充当前网站的域名，其他字段为空，用户可以在编辑器中完整配置。

**从右键菜单拾取 vs 从编辑器拾取**

| 对比项 | 右键菜单快速拾取 | 规则编辑器拾取 |
|--------|-----------------|---------------|
| **入口** | 右键 → 添加自定义规则 → 选择类型 | 设置 → 提取规则 → 新建/编辑 → 点拾取按钮 |
| **适用场景** | 快速创建、边看边配 | 完整配置、精细调整 |
| **编辑器界面** | 底部浮动提示栏（轻量） | 右侧抽屉（完整表单） |
| **增量拾取** | ✅ 支持，一次会话拾取多个字段 | ✅ 支持，每次点拾取按钮 |
| **实时测试** | 需点击"测试规则"按钮 | 自动实时预览 |
| **数据互通** | ✅ 草稿可随时打开完整编辑器继续编辑 | （同上） |

## 11. 导入与导出

### 11.1 导出格式

规则以 **JSON 格式**导出，文件扩展名为 `.json`，文件名形如 `flowx-rules-20260718.json`。

```json
{
  "version": "1.0",

    {
      "name": "我的博客规则",

      "matchValue": "example.com",
      "pathPattern": "",
      "titleSelector": ".post-title",
      "contentSelector": ".post-content",
      "bylineSelector": ".author-name",
      "dateSelector": ".publish-date",
      "siteName": "我的博客",

      "removeSelectors": [".ad", ".recommend"],

      "source": "manual"
```

### 11.2 导出功能

- **导出全部**：导出所有自定义规则
- **导出选中**：在规则列表中勾选多条，批量导出
- **导出单条**：每条规则的操作菜单中有"导出"选项

使用 Electron 的 `dialog.showSaveDialog` 让用户选择保存位置。

### 11.3 导入功能

| 导入模式 | 说明 | 冲突处理 |
|----------|------|----------|
| **合并导入**（默认） | 将导入的规则添加到现有规则列表中 | 匹配域名相同的规则自动跳过，并提示跳过数量 |
| **替换导入** | 清空现有自定义规则，替换为导入的规则 | 二次确认，防止误操作 |

**导入流程**

1. 用户点击"导入规则"按钮
2. 弹出文件选择对话框，选择 JSON 文件
3. 解析文件，校验格式和版本
4. 显示导入预览：共 X 条规则，其中 X 条新增，X 条重复将跳过
5. 用户确认后执行导入
6. 显示导入结果：成功导入 X 条，跳过 X 条

### 11.4 版本兼容

- 导出文件中包含 `version` 字段，标识规则格式版本
- 导入时检查版本号，低版本可自动升级，高版本提示用户升级应用

## 13. 技术坑点与解决方案

在实现用户自定义站点规则的过程中，遇到了几个典型的 Electron + Vue 技术坑点。本节记录这些问题及其解决方案，供后续开发参考。

### 13.1 IPC 对象克隆错误

**问题描述**

保存规则时出现 `An object could not be cloned` 错误，导致规则无法通过 IPC 从渲染进程发送到主进程。

**原因分析**

Vue 3 的响应式对象（`ref` / `reactive`）是 `Proxy` 代理对象，而 Electron 的 IPC 通信使用结构化克隆算法（Structured Clone Algorithm），**无法克隆 Proxy 对象**。直接将响应式对象通过 IPC 发送会导致克隆失败。

> ❌ **错误写法**
>
> ```typescript
> // 直接发送响应式对象会失败
> await electronApi.browser.saveCustomRule(form.value)
> ```

**解决方案**

使用 Vue 的 `toRaw()` 函数将响应式对象转换为原始对象后再发送。同时建议只提取需要的字段，避免传递不必要的响应式元数据。

> ✅ **正确写法**
>
> ```typescript
> import { toRaw } from 'vue'
>
> // 构建纯对象数据（去除响应式代理）
> function buildRuleData(): Partial<CustomSiteRule> & { contentSelector: string } {
>   const raw = toRaw(form.value)
>   const data: any = {}
>   // 只复制有效字段，跳过 undefined
>   const fields = [
>     'name', 'enabled', 'matchType', 'matchValue', 'pathPattern',
>     'contentTypes', 'contentSelector', 'titleSelector', 'bylineSelector',
>     'dateSelector', 'siteName', 'imageSelector', 'tagsSelector',
>     'removeSelectors', 'remark', 'source',
>   ]
>   for (const f of fields) {
>     const val = (raw as any)[f]
>     if (val !== undefined) {
>       data[f] = val
>     }
>   }
>   return data
> }
>
> // 发送纯对象
> await electronApi.browser.saveCustomRule(buildRuleData())
> ```

### 13.2 原生视图层级问题

**问题描述**

规则编辑器采用抽屉/弹窗形式时，会被浏览器页面（WebContentsView）遮挡，无论设置多高的 z-index 都无效。

**原因分析**

Electron 的 `WebContentsView` / `BrowserView` 是**原生窗口级别**的视图，直接由操作系统管理，不属于 Chromium 的渲染层级。渲染进程中的 DOM 元素（包括弹窗、抽屉、Tooltip）都在同一个合成层中，无法覆盖在原生视图之上。

> ⚠️ **Electron 视图层级**
>
> ```
> ┌─────────────────────────────────────┐
> │  原生窗口级别（操作系统管理）        │
> │  ┌───────────────────────────────┐  │
> │  │  WebContentsView（浏览器）    │  │  ← 在上层，无法被 DOM 覆盖
> │  └───────────────────────────────┘  │
> │  ┌───────────────────────────────┐  │
> │  │  渲染进程 DOM（HTML/CSS）     │  │  ← 在下层
> │  └───────────────────────────────┘  │
> └─────────────────────────────────────┘
> ```

**解决方案**

采用**并排布局 + 标签页切换**的方案，将规则管理界面与发布编辑面板整合到右侧面板中，从根本上避免 z-index 问题。

- **方案一（已采用）**：右侧面板内嵌编辑器，与浏览器并排显示
- **方案二**：使用 `BrowserWindow` 子窗口（但会有窗口焦点问题）
- **方案三**：在 WebContentsView 内部注入 UI（但跨域页面无法注入）

### 13.3 顶部导航栏下拉菜单层级问题

**问题描述**

顶部导航栏的 `el-sub-menu`（系统配置下拉）在浏览器页面会被 WebContentsView 遮挡，鼠标经过导航栏时需要隐藏浏览器视图，导致页面闪烁，体验很差。

**原因分析**

与 13.2 相同的根因：WebContentsView 是原生视图，层级高于 DOM。Element Plus 的下拉菜单是纯 HTML 实现，必然会被遮挡。

最初的临时方案是：监听导航栏的 `mouseenter` / `mouseleave` 事件，鼠标进入时将 WebContentsView 移到屏幕外（x:-9999），鼠标离开后再移回来。但这个方案有明显的体验问题：
- 鼠标快速扫过导航栏，浏览器会闪烁
- 下拉菜单展开过程中浏览器内容消失，视觉突兀
- 「关于」对话框等全局弹层也会触发浏览器隐藏

**解决方案：原生菜单替换 HTML 下拉**

将需要展开下拉的菜单项改为 Electron 原生 `Menu`，原生菜单由操作系统管理，层级最高，不会被 WebContentsView 遮挡。

**实现要点**：

1. **主进程新增 IPC**：`system:popupNativeMenu`，接收菜单项数组和坐标，调用 `Menu.buildFromTemplate` + `menu.popup()`
2. **菜单项改独立触发器**：用 `div` + 自定义样式替代 `el-menu-item`，脱离 `el-menu` 的事件系统，避免 `@select` 与 `@click` 冲突
3. **坐标计算**：用 `getBoundingClientRect()` 获取菜单项位置，在菜单项底部弹出菜单
4. **点击结果回传**：Promise 返回被点击的菜单项 id，渲染进程根据 id 执行路由跳转

> ✅ **原生菜单方案优势**
>
> | 对比项 | HTML 下拉（旧） | 原生菜单（新） |
> |--------|----------------|----------------|
> | 层级 | 被 WebContentsView 遮挡 | 原生菜单在最上层 |
> | 浏览器闪烁 | 每次 hover 都隐藏 | 完全不隐藏 |
> | 样式 | Element Plus 风格 | 系统原生风格 |
> | 交互响应 | 即时 | 即时 |

**延伸方案：全局对话框独立窗口化**

对于「关于」等全局对话框，同样会被 WebContentsView 遮挡，需要隐藏浏览器。更好的方案是将这类对话框改为独立的模态 `BrowserWindow`：

- 独立窗口层级最高，不会被 WebContentsView 遮挡
- 完全不需要隐藏浏览器，体验更流畅
- 设置 `modal: true` + `parent: mainWindow`，保持模态对话框的交互体验
- 路由增加 `meta.hideHeader` 标记，在 `App.vue` 中条件渲染顶部导航栏

**新增文件**：
- `src/main/windows/AboutWindow.ts` — 关于窗口主进程代码
- `src/renderer/pages/AboutWindow.vue` — 关于窗口页面组件

### 13.4 拾取器事件链路完整性

**问题描述**

通过右键菜单「添加自定义规则」启动拾取器时，渲染进程收不到拾取器启动事件，导致规则面板不显示、状态不同步。

**原因分析**

拾取器的启动有两条路径：

- **路径一**：用户在规则编辑器中点击「拾取」按钮 → 渲染进程主动调用 IPC → 状态同步
- **路径二**：用户右键菜单选择「添加自定义规则」 → 主进程直接启动拾取器 → 渲染进程未监听启动事件

路径二缺少 `pickerStarted` 事件的监听，导致渲染进程不知道拾取器已启动。

**解决方案**

在 preload 脚本和渲染进程中完整注册拾取器相关事件的监听器：

> ✅ **完整事件链路**
>
> ```typescript
> // preload/index.ts
> onPickerStarted: (cb) => {
>   const handler = (_event, payload) => cb(payload);
>   ipcRenderer.on('browser:pickerStarted', handler);
>   return () => ipcRenderer.removeListener('browser:pickerStarted', handler);
> },
> onPickerResult: (cb) => { ... },
> onPickerCancelled: (cb) => { ... },
>
> // Browser.vue - 挂载时注册监听
> onMounted(() => {
>   // 拾取器启动（外部触发时同步状态）
>   offPickerStarted = electronApi.browser.onPickerStarted((data) => {
>     if (data.viewId !== activeTabId.value) return
>     pickerActive.value = true
>     currentPickerField.value = data.fieldType
>     // 自动切换到规则面板
>     if (layoutMode.value === 'browser-only') {
>       layoutMode.value = 'split'
>     }
>     rightPanelTab.value = 'rules'
>   })
> })
> ```

### 13.5 响应式数据与 IPC 的最佳实践

基于以上坑点，总结以下 Vue + Electron IPC 开发最佳实践：

| 原则 | 说明 | 示例 |
|------|------|------|
| **toRaw 转换** | 发送 IPC 消息前，始终用 `toRaw()` 转换响应式对象 | `toRaw(form.value)` |
| **字段提取** | 只提取需要的字段，避免传递整个响应式对象 | 构建 data 对象，只包含有效字段 |
| **双向事件** | 主进程主动触发的操作，必须有对应的事件通知渲染进程 | pickerStarted / pickerResult / pickerCancelled |
| **视图并排** | 需要与 WebContentsView 交互的 UI，优先采用并排布局 | 右侧面板 + Tab 切换 |
| **viewId 校验** | 多标签页场景下，事件处理时始终校验 viewId 是否匹配当前标签 | `if (data.viewId !== activeTabId.value) return` |
