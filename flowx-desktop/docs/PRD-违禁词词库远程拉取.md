# PRD — 违禁词词库远程拉取（热更）

> 状态：Draft v1 ｜ 优先级：P1 ｜ 负责人：产品（PM）｜ 关联：合规预检 P0、本地覆盖文件（B）
> 最后更新：2026-07-16

---

## 1. 问题陈述（Problem Statement）

合规 P0 已上线，违禁词当前**写死在打包内的 JSON**（`src/main/resources/compliance/*.json`，按平台拆分如 `douyin.json`）。任何词库变更（平台规则更新、新黑话/梗、监管变动）都必须**改代码 → 走发版**才能到达用户。

- **谁遇到**：运营/合规负责人（要改词）、终端用户（因词库过期导致漏检/误检）、工程（被拉去改词出包）。
- **频率**：暂不确定，但违禁词属"政策驱动、变化频繁"内容，按"经常变"保守设计。
- **不解决的代价**：改词交付时延 = 发版周期（天级），运营无法自助；词库过期 → 漏检（违规风险）或误检（体验受损）；工程被词库变更持续打断。

---

## 2. 目标（Goals）

| # | 目标（可衡量） | 成功如何判定 |
|---|---|---|
| G1 | 改词**交付时延**从"发版周期"降至**后台改词后 ≤ 30 分钟全量生效** | 运营在后台改词 → 客户端下次刷新/启动即生效，无需出包 |
| G2 | 词库变更**运营自助率 ≥ 90%**（不再提工程工单） | 统计：因词库变更引发的工程工单数趋近于 0 |
| G3 | 远程拉取**成功率 ≥ 99%**（含缓存命中） | 监控：fetch 成功 / (fetch 尝试) 比率 |
| G4 | **零隐私回归**：匹配仍 100% 在本机，用户内容不上传 | 代码评审 + 抓包确认：仅 GET 词表，无内容外发 |

业务目标：把"改词"从工程排期摘出，缩短合规响应时延；为未来 B2B 多客户词库预留结构。

---

## 3. 非目标（Non-Goals）

| 非目标 | 为何本期不做 |
|---|---|
| ❌ 上传用户内容/发布文案到服务端做检测 | 违背"🔒 全部检测在本机完成"隐私承诺，且非必要 |
| ❌ 应用内管理 UI（增删词后台页面，方案 D） | 本期由运营在后端改词即可；B2B 按需再上 |
| ❌ 实时流式推送（WebSocket 下发） | 轮询/启动拉取 + 缓存足够，实时推送复杂度高、收益低 |
| ❌ 多租户实际serving（按客户返回不同词库） | 当前单一全局词库；本期仅**预留 tenant 维度**，不实现分租户逻辑 |
| ❌ 改动本地匹配引擎（Aho-Corasick） | 引擎稳定，仅替换"词表来源" |

## 4. 用户故事（User Stories）

优先级从高到低：

1. **作为运营/合规负责人**，我希望在后台改词后，客户端能在短时间内自动更新词库，以便无需等发版就能拦截新违规词。
2. **作为终端用户**，我希望打开/刷新应用时词库是最新的，以便减少因词库过期导致的漏检/误检。
3. **作为终端用户**，我希望在离线/网络异常时仍能用上一次的有效词库（或内置兜底），以便功能不中断。
4. **作为高级用户**，我希望能用本地文件追加自己的词或抑制个别误报词，以便适配个人场景（沿用本地覆盖文件 B）。
5. **作为运维**，我希望拉取失败/签名校验失败时有明确日志与兜底，以便快速定位问题。
6. **作为（未来）B2B 客户管理员**，我希望我的词库与其他客户隔离，以便合规策略不串味（本期仅预留，不实现）。

边界/异常：
- 拉取超时、HTTP 非 2xx、签名校验失败 → 不覆盖现有词库，回退缓存/内置。
- 首次安装无缓存 → 直接用内置兜底，后台静默重试拉取。

## 5. 需求（Requirements）

### P0 — Must Have（v1 必须）

**R1 启动拉取 + 定时刷新**
- 应用启动后（合规模块初始化时）调用接口拉取词库；并支持**周期性刷新**（默认每 6 小时一次）与**手动"检查更新"入口**。
- 接受准则：Given 应用启动，When 网络可用，Then 客户端拉取成功并加载最新词库；离线时回退。

**R2 版本化 + 304 协商**
- 请求携带已缓存 `version`；服务端无变更返回 `304 Not Modified`，客户端沿用缓存。
- 接受准则：Given 缓存版本=最新，When 拉取，Then 返回 304，不下载整包，不重置缓存。

**R3 签名校验（防篡改）**
- 词库包带签名；客户端用**内置公钥**验签，失败则拒绝加载并回退。
- 接受准则：Given 词库被篡改/签名不符，When 拉取，Then 拒绝加载、保留上一有效词库、记警告日志。

**R4 本地缓存 + 离线兜底**
- 拉取成功并验签通过后写入 `userData/cache/compliance/<platform>.json`；加载顺序：**缓存(验签通过) → 内置兜底**。
- 接受准则：Given 完全离线且无缓存，When 启动，Then 加载内置 JSON，功能可用，不崩。

**R5 与本地覆盖文件（B）协同的优先级**
- 生效优先级（低→高）：内置兜底 < **远程拉取(管理源，权威)** < 本地覆盖文件(用户级追加/抑制)。
- 本地覆盖文件可**追加词**与**抑制(排除)特定管理词条**（解决误报）。
- 接受准则：Given 远程词库含"全网最低"、本地覆盖文件排除该词，When 扫描，Then 不命中该词。

**R6 tenant 维度预留**
- 请求参数与本地存储结构预留 `tenant` 字段；本期固定 `tenant=default`，服务端按 `default` 返回。
- 接受准则：Given 本期，When 拉取，Then `tenant=default`，不影响现有逻辑；未来改值即可分租户。

### P1 — Should Have（快跟）

- **R7 手动"检查更新"按钮 + 状态提示**（上次更新时间、当前版本、是否最新）。
- **R8 拉取失败可观测**：metrics 上报 fetch 成功/失败/回退次数（脱敏，不含词内容）。
- **R9 密钥轮换预案**：支持内置公钥多版本/兜底，避免单点失效（文档化，不强制实现轮换 UI）。

### P2 — Future（架构保险，本期不做）

- **R10 多租户 serving**：服务端按 `tenant` 返回不同词库，客户端按登录/配置切换。
- **R11 应用内管理 UI（方案 D）**：客户自助增删词。
- **R12 回滚机制**：服务端一键回退上一稳定版本，客户端拉取即生效。

## 6. 接口契约草案（API Contract）

> 产品侧定义字段与语义；工程据此实现。Endpoint 域名/路径待定（Open Question）。

**请求**
```
GET {BASE}/compliance/dictionary?platform=douyin&version=20260715.1&tenant=default
Authorization: Bearer <客户端凭证，仅用于配额/审计，不含用户内容>
```

**响应 — 有更新（200）**
```json
{
  "platform": "douyin",
  "tenant": "default",
  "version": "20260716.1",
  "updatedAt": "2026-07-16T09:00:00Z",
  "algorithm": "aho-corasick",
  "signature": "<base64(私钥签名 of canonical payload)>",
  "entries": {
    "high": ["绝对化用词A", "..."],
    "mid":  ["全网最低", "..."],
    "low":  ["...", "..."]
  }
}
```

**响应 — 无更新（304）**：空 body，客户端沿用缓存。

**校验流程**：客户端用内置公钥对 `canonical(payload)` 验 `signature` → 通过则写缓存并加载；失败则丢弃并回退。

## 7. 成功指标（Success Metrics）

**Leading（上线后 1–4 周）**
- 拉取成功率 ≥ 99%（含 304/缓存命中）
- 缓存命中率（离线/304 占比）
- 后台改词 → 客户端生效**中位时延 ≤ 30 分钟**（按版本号传播测算）
- 手动"检查更新"点击率（验证入口有用）

**Lagging（1 季度）**
- 因词库过期导致的漏检/误检工单下降（对比 P0 上线前基线）
- 词库变更运营自助率 ≥ 90%（工程工单≈0）
- 合规相关支持工单量变化

**测量方式**：客户端匿名上报 fetch 结果计数（成功/失败/回退/版本号），不含任何词内容或用户内容；后台看板统计。
