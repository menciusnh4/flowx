# 小红书自动发布功能 - 技术实现文档

## 概述

本文档记录 FlowX 桌面端「小红书一键发布」功能的完整技术实现，包括页面结构分析、遇到的核心问题、解决方案和关键代码路径。

---

## 一、目标页面结构分析

### 1.1 发布页 URL

```
https://creator.xiaohongshu.com/publish/publish
```

### 1.2 页面核心结构（DOM + Shadow DOM）

页面上半部分（上传 + 标题 + 正文）是标准 DOM：

```html
<!-- 文件上传区域 -->
<div class="upload-wrapper">
  <input type="file" class="upload-input" style="visible: false" />
  <button class="d-button">上传视频</button>
</div>

<!-- 标题输入 -->
<input type="text" placeholder="填写标题会有更多赞哦" />

<!-- 正文输入 -->
<div contenteditable="true" style="width: 632px; height: 140px"></div>
```

页面底部（发布按钮）是 **Closed Shadow DOM 自定义组件**，**外部 JS 无法穿透**：

```html
<xhs-publish-btn
  data-v-1e3023c7=""
  data-v-74e5df5a-s=""
  is-publish="true"
  is-save-draft="true"
  submit-text="发布"
  save-text="暂存离开"
  submit-disabled="true"   <!-- 禁用时为 "true"，可用时为 "false" -->
  save-disabled="false">
  # shadow-root (closed)   ← JS 无法访问
    <div class="publish-page-publish-btn">
      <button class="ce-btn white">暂存离开</button>   <!-- 左：草稿按钮 -->
      <button class="ce-btn bg-red">发布</button>       <!-- 右：真正的发布按钮 -->
    </div>
</xhs-publish-btn>
```

### 1.3 关键属性说明

| 属性 | 类型 | 说明 |
|------|------|------|
| `submit-disabled` | `"true"` / `"false"` | 发布按钮是否可用。**按钮在视口外时永远为 "true"，即使内容已填好** |
| `submit-text` | string | 发布按钮的文字（固定为 `"发布"`） |
| `save-text` | string | 草稿按钮的文字（固定为 `"暂存离开"`） |

### 1.4 发现的两个重要行为

1. **按钮只有在视口内才会检测表单完整性** → 程序操作时必须先 `scrollIntoView()`，否则 `submit-disabled` 永远不会变为 `"false"`
2. **Shadow DOM 内部的 `<button class="ce-btn bg-red">发布</button>` 无法通过 JS `querySelector` 找到** → 必须用 **Chrome DevTools Protocol (CDP)** 穿透

## 二、完整发布流程

```
1. 导航到 https://creator.xiaohongshu.com/publish/publish
2. 等待页面加载完成（约 5-6 秒）
3. 验证登录状态（cookies 已注入，页面显示创作服务平台）
4. 文件上传：
   a. CDP DOM.getDocument(pierce:true) 查找 input[type=file]
   b. DOM.setFileInputFiles(nodeId, [视频文件路径])
   c. 等待上传处理完成（上传状态通过 visible inputs 检测）
5. 填写标题：
   a. evalJS 查找 placeholder="填写标题会有更多赞哦" 的 input
   b. 用原生 setter 设置 value
6. 填写正文：
   a. evalJS 查找 contenteditable 元素
   b. 通过 innerText 设置
7. 点击发布按钮（见第三节详细说明）
8. 发布后验证：检测 URL 是否跳转到 /publish/success
```

## 三、核心难点与解决方案

### 难点 1：上传视频 — CDP file chooser

**问题**：`file input` 不可见（`visible: false`），普通 Web API 无法操作。

**解决方案**：使用 Chrome DevTools Protocol 直接操作 DOM 节点：

```
CDP.DOM.getDocument({ pierce: true, depth: -1 })
  → 在返回树中找 nodeName: "INPUT" 且 attributes 含 type="file"
  → CDP.DOM.setFileInputFiles({ nodeId: xxx, files: [videoPath] })
```

**代码位置**：`src/main/services/PlatformAdapter.ts`，约 1420-1450 行的上传策略部分

**日志关键字**：`策略 1：CDP 直接扫描 file input 并注入` → `✅ file input #0 注入成功`

---

### 难点 2：发布按钮禁用 + 不在视口 → 13 秒超时

**问题**：`xhs-publish-btn` 在视口外时 `submit-disabled="true"`，组件内部的 Vue watcher 只在 `scroll` 事件触发时重新检查表单完整性。程序运行时页面不滚动，按钮永远禁用。

**原始日志（修复前）**：
```
发布按钮禁用 (submit-disabled=true)，等待…  ✖ 13 次 × 800ms = 10+ 秒
✅ 发布按钮就绪                       ⇦ 滚动后才变为 false
```

**解决方案**：每次轮询前先调用 `element.scrollIntoView({ block: "center" })`，再检查属性：

```javascript
// 轮询逻辑（伪代码）
while (timeout < 15000ms) {
  const result = await page.evaluate(() => {
    const el = document.querySelector('xhs-publish-btn');
    if (!el) return { status: 'not_found' };
    el.scrollIntoView({ block: 'center', behavior: 'instant' }); // ← 关键
    const disabled = el.getAttribute('submit-disabled');
    const rect = el.getBoundingClientRect();
    return {
      status: disabled === 'true' ? 'disabled' : 'ready',
      x: rect.left + rect.width * 0.75,  // 右 75% = 发布按钮
      y: rect.top + rect.height / 2,
    };
  });
  if (result.status === 'ready') {
    targetX = result.x;
    targetY = result.y;
    break;
  }
  await sleep(400); // 轮询间隔
}
```

**效果对比**：

| 阶段 | 修复前 | 修复后 |
|------|--------|--------|
| 从填写完成到按钮可用 | ~13 秒 | ~1-2 秒 |
| 发布总时长 | ~28 秒 | ~15 秒 |

**代码位置**：`src/main/services/PlatformAdapter.ts`，约 1483-1520 行

**日志关键字**：`✅ 发布按钮就绪` / `发布按钮禁用`

---

### 难点 3：Closed Shadow DOM — 普通 JS 找不到内部按钮

**问题**：`<xhs-publish-btn>` 的 shadow root 是 closed 模式，`element.shadowRoot` 返回 `null`。用 JS 定位不到内部的 `<button class="ce-btn bg-red">`。

**尝试过但失败的方案**：

| 方案 | 结果 | 原因 |
|------|------|------|
| `document.querySelector('.ce-btn')` | ❌ null | 在 shadow DOM 内，外部选不到 |
| 在 host 元素上 `el.click()` | ❌ 无效 | click 事件需要真正的按钮元素响应 |
| `dispatchEvent(new MouseEvent('click', {bubbles:true}))` | ❌ 无效 | 组件内部不监听 host 元素的事件 |
| PointerEvent + MouseEvent 连发 | ❌ 无效 | 同上 |

**最终方案（有效）**：CDP 协议穿透 shadow DOM + 坐标点击

```
CDP.DOM.getDocument({ pierce: true, depth: -1 })
  └→ 遍历所有节点：nodeName === "BUTTON" || nodeName === "A" || nodeName === "INPUT"
     └→ 对每个候选节点调用 CDP.DOM.getBoxModel({ nodeId })
        └→ 筛选：
            (a) y 在 host 元素范围 [771, 771+90]
            (b) x 在 host 右半部分 [272+680*0.45, 272+680]
            (c) class 含 "red" / "publish" / "submit" / "ce-btn"
           └→ 得到真实的 <button class="ce-btn bg-red"> 坐标 (684, 816)
              └→ CDP.Input.dispatchMouseEvent: mouseMoved → mousePressed → mouseReleased
              └→ 补充 CDP DOM.focus(nodeId) + dispatchKeyEvent(Enter)
```

**实际命中的按钮信息**（日志验证）：
```
shadow DOM 中共搜到 59 个 BUTTON/A/INPUT 节点
右半部分候选按钮: 1 个 → [{x:684, y:816, class:"ce-btn bg-red", tag:"BUTTON", nodeId:2344}]
✅ 选定 shadow DOM 内部按钮 {nodeId: 2344, x: 684, y: 816, class: "ce-btn bg-red"}
```

**代码位置**：`src/main/services/PlatformAdapter.ts`，约 1538-1620 行

**日志关键字**：`shadow DOM 中共搜到` / `右半部分候选按钮` / `✅ 选定 shadow DOM 内部按钮` / `✅ 合成鼠标点击` / `✅ Enter键点击`

---

### 难点 4：发布按钮坐标 — 左右两个按钮布局

**问题**：host 元素 `xhs-publish-btn` 的中心（50%）正好命中左边的"暂存离开"按钮（草稿），点了后内容被保存为草稿，永远不发布。

**DOM 结构（物理坐标）**：

```
host rect: left=272, top=771, width=680, height=90

  left=272                    center=612                 right=952
     |                           |                         |
     ┌───────────────────────────┬──────────────────────────┐
     |       暂存离开            |        发布（bg-red）    | ← y=816 (center)
     |     ce-btn white          |      ce-btn bg-red       |
     └───────────────────────────┴──────────────────────────┘
     |<--- 340px (左 50%) --->|<--- 340px (右 50%) --->|
         ↑ 50% 位置: 612 → 草稿!         ↑ 75% 位置: 782 → 真正的发布按钮!
```

**解决方案**：点击坐标固定为 host 元素 **x + width \* 0.75**，y = center

**代码位置**：轮询 JS 中的 `x:Math.round(r.left+r.width*0.75)`

---

### 难点 5：误点击"发布笔记"导航按钮

**问题**：早期版本用模糊文本匹配找"发布"相关元素。页面顶部导航含"发布笔记"文字（一个 div），它的 y 坐标在视口内，被误判为发布按钮。

**修复**：
1. 找按钮时**必须排除** tag 为 div/span 的文本元素
2. 只选 BUTTON/A/INPUT 节点（通过 CDP DOM 遍历）
3. 用 BoxModel 精确筛选「右半部分 + host y 范围」的按钮
4. 按 class 关键词加权：`ce-btn` + `bg-red` + `publish` = 最高优先级

## 六、排错指南

如果某次发布失败，按以下顺序排查日志：

### Step 1：看日志里是否有这 3 个关键字

```
✅ 发布按钮就绪            ← 找到了 xhs-publish-btn 且 submit-disabled="false"
✅ 选定 shadow DOM 内部按钮 ← 穿透 closed shadow DOM 拿到了真正按钮的坐标
verdict: "success"         ← 点击后页面正确跳转
```

### Step 2：常见失败模式

| 失败现象 | 可能原因 | 检查日志 |
|----------|----------|----------|
| `发布按钮禁用` 一直循环到 15 秒超时 | 视频没上传完成 / 标题或正文为空 | 检查 `poll-content-fields` 返回的 visibleInputs：标题 input 是否有值 |
| `shadow DOM 中共搜到 0 个` | CDP DOM 搜索未打开 pierce:true，或页面结构变化 | `诊断：xhs-publish-btn 元素` — 看 rect 和 attributes |
| `右半部分候选按钮: 0 个` | 发布按钮 class 改了（不再是 ce-btn bg-red） | 把 search 放宽或看 `shadow DOM 中共搜到` 的每个元素 class |
| 点击后停留在 `/publish/publish`，URL 没变 | 点错了按钮（可能 hit 了"暂存离开"）| 看 `选定 shadow DOM 内部按钮` 的坐标是否在右侧 |
| verdict: unclear 但手动点击能成功 | 点击坐标偏差 > 按钮尺寸 | 看 BoxModel 返回的 y 是否超出 host range |

### Step 3：应急方案

如果后续小红书更新 UI 导致发布按钮无法自动点击，FlowX 会：

```
主进程日志输出：
  ⚠️ 未找到发布按钮，请在打开的窗口中手动点击

用户在浏览器窗口中点击红色"发布"按钮 → 流程继续
```

## 七、未来可能的变更点

1. **`xhs-publish-btn` 组件**：如果 class 从 `ce-btn bg-red` 改变，需要更新 CDP 搜索中的 class 关键词
2. **`submit-disabled` 属性名**：如果改名为 `disabled` 或 `is-disabled`，需要更新轮询的 `getAttribute()` 部分
3. **上传区域 HTML 结构变化**：需要调整 `input[type=file]` 探测逻辑
4. **CDP 协议版本**：CDP `DOM.getDocument` 中 `pierce` 参数在未来 Chromium 中可能改名

建议每次大版本更新小红书后，先跑一次「一键发布」测试，关注以下日志行：
- `诊断：xhs-publish-btn 元素` — 确认 host 元素属性
- `shadow DOM 中共搜到 N 个 BUTTON/A/INPUT 节点` — 确认搜索数量合理
- `✅ 选定 shadow DOM 内部按钮` — 确认命中的是 `class: "ce-btn bg-red"` 的 BUTTON
- `verdict: "success"` — 确认发布跳转

## 八、架构演进：从 getAdapter 到 PlatformDispatcher（工厂方法模式）

### 8.1 背景

最初的发布流程是"单方法发布":

```typescript
PlatformAdapter.getAdapter('xiaohongshu').publish(accountId, req, onProgress)
```

问题:
- 所有内容类型（视频/图文/文章）走同一 `publish` 方法，内部用 `if contentType === 'video'` 分支，违背单一职责
- 平台之间逻辑耦合：一个平台修改可能影响其他平台
- 调试困难：无法明确知道当前执行的是哪类内容的发布逻辑

### 8.2 新架构：工厂方法模式 + 适配器模式

```typescript
// 1. 平台适配器接口扩展（PlatformAdapter 接口在 types.ts）
interface PlatformAdapter {
  key: string;
  meta: PlatformMeta;
  capabilities: AccountCapabilities;
  detectLoggedIn(win: BrowserWindow): Promise<LoginCheckResult>;
  extractPageInfo?(win: BrowserWindow): Promise<ExtractedAccountInfo>;
  publishVideo?(accountId, req, onProgress): Promise<PublishItemProgress>;
  publishImage?(accountId, req, onProgress): Promise<PublishItemProgress>;
  publishArticle?(accountId, req, onProgress): Promise<PublishItemProgress>;
  publish?(accountId, req, onProgress): Promise<PublishItemProgress>;  // 兼容接口
}

// 2. PlatformDispatcher（工厂方法）— 两级分发
export function createExecutor(
  platform: string,
  contentType: 'video' | 'image' | 'article',
): PublishExecutor {
  const adapter = getPlatformAdapter(platform);
  // 优先调用精确方法，降级到通用 publish
  switch (contentType) {
    case 'video':   return { execute: adapter.publishVideo || adapter.publish };
    case 'image':   return { execute: adapter.publishImage || adapter.publish };
    case 'article': return { execute: adapter.publishArticle || adapter.publish };
  }
}

// 3. PublishEngine 使用工厂方法
const executor = PlatformDispatcher.createExecutor(platform, contentType);
return executor.execute(accountId, request, onProgress);
```

### 8.3 对小红书的影响

- 原 `xiaohongshu.ts` 只实现 `publish`，现在拆分为 `publishVideo` 和 `publishImage`
- 两套独立的配置（视频 URL vs 图文 URL，不同的上传文件类型）
- 两个方法都共享 `shared.ts` 的 `runStandardPublish`，传入不同的 config
- 第三方的按钮点击、CDP 穿透 shadow DOM 等技术实现完全隔离，不影响抖音/快手
