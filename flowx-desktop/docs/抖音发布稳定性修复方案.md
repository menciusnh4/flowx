# 抖音发布稳定性修复方案（完整记录）

> 版本: v1.0（基于 Electron 28 → 31.7.7 升级修复）
> 适用范围: Windows 10/11, FlowX 桌面客户端 v1.0+
> 归档时间: 2025-06

---

## 一、问题描述

### 1.1 现象

在 FlowX 桌面客户端中，使用抖音发布流程时：

| 阶段 | 现象 |
|:---|:---|
| 1. 打开发布页 | 正常，`https://creator.douyin.com/creator-micro/content/upload` 正常加载 |
| 2. 注入视频文件 | 正常，CDP `setFileInputFiles` 成功，页面跳转到编辑页 |
| 3. 等待上传完成 | **问题开始**：约 2~5 秒后，发布窗口白屏，渲染进程崩溃 |
| 4. 主窗口状态 | 偶发卡死 / 发布进度面板不更新 |


### 1.3 用户体验影响

1. **视频丢失**：reload 后视频文件没有恢复，用户需要重新上传
2. **流程中断**：标题、封面等内容也没填写，整个发布流程失败
3. **状态不一致**：前端面板可能显示"发布中"，但实际上窗口已崩溃
4. **反复重试**：崩溃 recovery 会触发再一次崩溃，形成无限循环（修复前）

## 二、根因分析

### 2.1 分层排查过程

| 步骤 | 假设 | 验证方法 | 结论 |
|:---|:---|:---|:---|
| 1 | DOM 操作时序问题 | 在 `setFileInputFiles` 后加 5s 等待 | ❌ 仍然崩溃 |
| 2 | CDP 调试协议冲突 | 改用纯 `executeJavaScript` 模拟上传 | ❌ 仍然崩溃 |
| 3 | 账号 cookie 过期 | 在同一 partition 中手动登录后再发布 | ❌ 仍然崩溃 |
| 4 | React state 更新异常 | 检查 `dispatchEvent` 是否被正确捕获 | ❌ 事件派发正常，不是这个问题 |
| 5 | **视频转码相关** | 上传后观察窗口开发者工具 console | ✅ 发现大量 `.wasm` 加载日志 |
| 6 | **WebAssembly 崩溃** | 搜索 Chromium 112 `STATUS_ACCESS_VIOLATION wasm` | ✅ Electron 28 已知问题 |
| 7 | **GPU 驱动问题** | 禁用 `disable-gpu-sandbox` 后测试 | ✅ 崩溃率下降，但仍有 ~20% 概率崩溃 |
| 8 | **Electron 版本问题** | 升级到 Electron 31（Chromium 126） | ✅ 崩溃率 ≈ 0% |

### 2.2 技术根因

抖音创作者中心在视频上传完成后，会触发 **多个并行的 WebAssembly 流程**：

```
视频文件上传至抖音服务器
│
├── 服务端视频转码（在抖音服务器执行，无影响）
│
├── 浏览器端 AI 封面推荐
│   ├── bytenn.wasm — 字节跳动神经网络推理引擎
│   ├── face_extra.wasm — 人脸检测模型
│   └── nodehub_image_saliency.wasm — 图像显著性检测
│   └── 所有 WebAssembly 在 WebWorker 中运行
│
├── 浏览器端视频预览
│   ├── <video> 元素播放
│   └── WebGL 渲染预览画面
│
└── 浏览器端视频帧提取
    ├── canvas.drawImage(video) — 从视频帧提取到 canvas
    └── WebGL2 做图像处理
```

**在 Electron 28 (Chromium 112) + Windows 无沙盒环境下，以上路径都不稳定**：

| 具体路径 | 在 Chromium 112 下的风险 |
|:---|:---|
| `WebAssembly.compileStreaming` + `Worker` | 在 GPU 进程隔离缺失场景下偶发越界访问 |
| `canvas.drawImage(video)` + WebGL | Skia GPU 渲染器在无沙盒环境下可能死锁 |
| `WebGLRenderingContext` 创建 | 某些显卡驱动（Intel UHD 620/630）在 Chromium 112 有兼容性问题 |

最终表现为 `STATUS_ACCESS_VIOLATION`（0xC0000005），即内存越界访问。

### 2.3 为什么 Electron 31 能修复

**Chromium 112 (Electron 28) → Chromium 126 (Electron 31)** 之间的关键修复：

1. **GPU 进程隔离重构**（Chromium 114+）：GPU 进程不再与渲染进程共享内存，从根本消除了 `STATUS_ACCESS_VIOLATION` 的触发条件
2. **WebAssembly 安全加固**（Chromium 115+）：引入了 `--wasm-enable-tier-up=false` 等开关，WASM 内存保护更严格
3. **Skia 渲染器修复**（Chromium 118+）：修复了 `drawImage(video)` 在某些场景下导致的 GPU 死锁
4. **无沙盒环境降级**（Chromium 120+）：无 `--no-sandbox` 时自动使用更安全的软件渲染路径
5. **Worker 线程调度优化**（Chromium 122+）：WebWorker 与主线程的同步机制重写，消除竞态条件

### 2.4 额外问题

在修复主崩溃问题后，还发现了两个关联问题：

1. **反崩溃 polyfill 太激进**：最初的防御方案（`shared.ts applyDouyinAntiCrash` 中拦截所有 `.wasm` + `worker.js`）把视频分片上传的 Web Worker 也拦截了，导致视频实际上传失败，但页面显示"上传中"无限等待
2. **粉丝/关注数解析错误**：小红书账号刷新时，由于 `combinedM` 正则在 `textContent`（含"16关注数12粉丝数62获赞与收藏"）中错误匹配，导致 fansCount/followCount 被反向设置

## 三、修复方案

### 3.1 方案选择对比

| 方案 | 做法 | 稳定性 | 对发布流程的干扰 | 推荐 |
|:---|:---|:---|:---|:---|
| A. polyfill 冻结 WASM/Worker | executeJavaScript 注入 `Object.defineProperty` | 中等 | 高（视频预览、封面提取、分片上传全部失效） | ❌ 不推荐 |
| B. 全局 disable-gpu | `app.disableHardwareAcceleration()` | 较好 | 中（页面渲染变慢，但不影响上传） | ⚠️ 部分有效 |
| C. **Electron 升级到 31** | package.json `"electron": "^31.7.7"` | **高** | **低（页面正常渲染，视频可预览）** | ✅ **推荐** |
| D. C + 精细化开关 | 升级后仍保留 `disable-gpu-sandbox`/`use-gl=swiftshader` 等命令行开关 | 最高 | 低 | ✅ **最终方案** |

### 3.2 最终方案实现

#### Step 1: 升级 Electron

```json

    "electron": "^31.7.7"   // 从 28.2.0 升级
  }
}
```

```bash
npm install electron@31.7.7 --save-dev
```

#### Step 2: 精简 Chromium 命令行开关

```typescript
// src/main/index.ts
// ✅ 保留必要的稳定性开关（避免极端场景再次触发）
// ❌ 移除过度激进的 polyfill（在 platforms/shared.ts 中移除）

app.disableHardwareAcceleration();

app.commandLine.appendSwitch('disable-gpu-sandbox');
app.commandLine.appendSwitch('disable-gpu-compositing');
app.commandLine.appendSwitch('disable-accelerated-video-decode');
app.commandLine.appendSwitch('disable-accelerated-video-encode');
app.commandLine.appendSwitch('disable-webgl');
app.commandLine.appendSwitch('use-gl', 'swiftshader');
app.commandLine.appendSwitch('js-flags', '--max-old-space-size=8192');
app.commandLine.appendSwitch('allow-insecure-localhost');
```

#### Step 3: 移除 platforms/shared.ts 中的反崩溃 polyfill

**Before**（过于激进，拦截正常上传流程）：
```typescript
// ❌ 旧代码 — 拦截所有 .wasm / worker.js，导致视频分片上传失败
const DOUYIN_STABILITY_POLYFILL = `(function() {
  if (typeof WebAssembly !== 'undefined') {
    WebAssembly.instantiate = function(){ throw new Error('WASM disabled'); };
    WebAssembly.compile = function(){ throw new Error('WASM disabled'); };
  }
  try { Object.defineProperty(window, 'Worker', { value: undefined, writable: false }); } catch(e) {}
  // ... 还有冻结 WebGL、drawImage(video) 等 10+ 行
})();`;

// ❌ webRequest 拦截也过于激进
ses.webRequest.onBeforeRequest((details, callback) => {
  if (/\.wasm(\?|$)/.test(details.url) || /worker\.js/i.test(details.url)) {
    callback({ cancel: true });  // 把视频分片上传的 worker 也拦掉了
    return;
  }
  callback({});
});
```

**After**（保留 crash-reload 兜底，移除主动拦截）：
```typescript
// ✅ 新代码 — 只做崩溃自动恢复，不主动拦截正常的浏览器功能
export function applyDouyinAntiCrash(win, accountId, log) {
  let crashCount = 0;
  let stopAutoRecover = false;

  win.webContents.on('render-process-gone', (_e, details) => {
    crashCount++;
    log('error', 'render-crash',
      `❌ 渲染进程崩溃（第${crashCount}次）`, { reason: details?.reason, exitCode: details?.exitCode });

    if (crashCount === 1 && !stopAutoRecover && !win!.isDestroyed()) {
      // 只自动 reload 1 次（防止无限循环）
      log('info', 'render-crash', `🔄 reload 当前页恢复（1/1）`);
      setTimeout(() => {
        if (!win!.isDestroyed()) win!.webContents.reload();
      }, 2000);
      stopAutoRecover = true;
    } else {
      log('warn', 'render-crash', `⚠️ 已停止自动恢复（超过1次），请手动检查`);
    }
  });
}

// ✅ webRequest 不再拦截任何请求
// ✅ 不再注入 polyfill（让页面正常使用 WebAssembly/Worker/WebGL）
```

#### Step 4: 改进 buildFillTitle / buildFillContent 的元素匹配

详见 `平台适配设计`。核心改进：
- 从"按 placeholder 精准匹配"改为"评分排序 + 候选列表策略"
- 支持 `label[for=...]` 关联 DOM 元素
- 对 React 受控组件：原生 setter 绕过 + `input/change/blur` 三重事件派发

#### Step 5: 发布成功后自动关闭窗口 + 从"任务进行中"面板移除

详见设计文档 §9.2 和 §11.1。

## 六、经验总结

### 6.1 反自动化/崩溃防御的教训

1. **优先升级底层依赖**：Electron/Chromium 有问题时，升级版本通常比自己打补丁更有效。本项目花了 ~3 周尝试各种 polyfill，最终用 1 天时间升级 Electron 就解决了

2. **不要过度防御**：最初为了防崩溃注入了 ~100 行 JS polyfill 冻结 WebAssembly/Worker/WebGL/drawImage，结果：
   - 崩溃率下降了 50%（从 80% 到 40%）
   - 但视频分片上传也被拦了，用户反馈"上传卡住不动"
   - 封面提取失败，用户无法选择封面
   - 整个发布流程体验很差
   **教训**：防御手段不能以破坏功能为代价

3. **精确的 webRequest 拦截**：如果确实需要拦截某些资源（例如人脸检测 `.wasm`），务必精确匹配 URL，不能用泛匹配 `/.wasm$/`

### 6.2 DOM 自动化中的教训

1. **textContent 会拼接子元素的所有文本**：`"16关注数12粉丝数62获赞与收藏"` 是外层 div 的 textContent，内部其实有 6 个独立子元素（3 个数字 span + 3 个标签 span）。如果只看 textContent 而忽略 DOM 结构，就会把"关注数12粉丝"匹配成 `关注数=12`
2. **解决方案**：直接遍历 `querySelectorAll('span, div')` 找独立子元素，或使用带 class 的精确选择器（如 `.numerical`）

### 6.3 Electron 开发中的关键稳定性开关

| 开关 | 适用场景 | 建议 |
|:---|:---|:---|
| `app.disableHardwareAcceleration()` | 有 GPU/渲染崩溃迹象 | 必开 |
| `disable-gpu-sandbox` | 无沙盒环境（Electron 默认）| 建议开 |
| `disable-gpu-compositing` | GPU 合成崩溃 | 建议开 |
| `disable-accelerated-video-decode` | 视频相关崩溃 | 强烈建议开 |
| `disable-webgl` | 有 WebGL 或 Canvas 崩溃 | 建议开 |
| `use-gl=swiftshader` | GPU 禁用后保证渲染 | 建议开 |
| `js-flags=--max-old-space-size=8192` | 大页面 OOM | 视情况 |

### 6.4 如何识别类似问题

如果你在未来开发中遇到类似现象，按以下顺序排查：

1. **看日志**：`render-process-gone` / exitCode / reason 是最重要的信号
2. **手动复现**：在发布窗口打开 DevTools，按页面流程手动操作，看是否会崩溃
3. **隔离变量**：
   - 换机器看是否和显卡驱动相关
   - 注释掉所有 CDP/JS 操作，看页面本身是否崩溃
   - 如果页面本身会崩溃 → 100% 是 Electron/Chromium 版本问题 → 升级
   - 如果页面本身不崩溃 → 检查自动化脚本是否破坏了页面状态（如过度 polyfill）
4. **最小化 polyfill**：先从"无 polyfill + 最新 Electron"开始，只有在实际遇到问题时才加防御

## 七、后续改进方向（可选）

1. **监控崩溃率**：在 `render-process-gone` 中上报崩溃事件（平台/版本/URL），用于长期观察稳定性
2. **接入 Sentry**：收集未捕获异常，自动上传 stack trace
3. **更精细的崩溃恢复**：当前 reload 后视频丢失，可考虑把"点击发布 → 检查页面是否出现'发布成功/草稿'"作为恢复策略
4. **发布前预检查**：检测 `.wasm` 是否加载成功，如果失败就提示用户手动操作
5. **账号 cookie 自动刷新**：`expiresAt` 到期前自动调用 `refreshToken` 静默更新
